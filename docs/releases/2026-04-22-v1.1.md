# v1.1.0-rc.1 (2026-04-22)

Phase 2 release candidate. Three new capabilities layered on top of v1.0's Agent SDK daemon. Full specs: `docs/specs/2026-04-22-{permission-relay,voice,companion}.md`. Unified plan: `docs/plans/2026-04-22-phase2-v1.1.md`.

## Highlights

- **`wechat-cc run --dangerously` revived**: reactive sessions run in `bypassPermissions` mode (matches `claude --dangerously-skip-permissions`). Phase 1 permission relay preserved for strict-mode use — picked per daemon launch.
- **Outbound voice**: `reply_voice` tool lets Claude reply with audio when explicitly asked. Primary provider is local **VoxCPM2** via `vllm serve --omni` (OpenAI-compatible `/v1/audio/speech`); Qwen DashScope retained as cloud fallback. Config entirely via WeChat conversation — no env vars, no editing files.
- **Companion layer** (default-off): opt-in proactive pushes via cron-scheduled Claude tasks. Two user-pickable **人格**:
  - **小助手** (assistant) — work-focused, strict push rules
  - **陪伴** (companion) — warmer, lighter rules
  - Users can edit `personas/*.md` to tune, or drop in a third custom persona
- **Configure-by-conversation pattern** established: `save_voice_config` + `companion_enable` + future Phase 3/4 tools reuse the same "Claude walks the user through setup in WeChat" shape.

## Breaking changes

None for existing v1.0 users who don't opt into voice or Companion. Daemon starts in strict mode by default; adding `--dangerously` or calling `companion_enable` are explicit choices.

Under `~/.claude/channels/wechat/`:
- `voice-config.json` (mode 0600) — TTS provider + base_url/api_key
- `companion/profile.md` — user profile (hand-editable)
- `companion/personas/{assistant,companion}.md` — persona definitions (hand-editable)
- `companion/config.json` — enabled flag, per-project persona, trigger registry
- `companion/runs.jsonl` — per-eval audit (rotates at 10 MB)
- `companion/push-log.jsonl` — push events (rotates at 10 MB)

All Phase 1 state files preserved unchanged.

## Internals

- **New deps**: `croner@^9` (cron parsing). No other runtime deps added.
- **New modules**: `src/daemon/tts/` (TTSProvider interface + Qwen + http-tts + voice-config), `src/daemon/companion/` (paths, templates, persona parser, config store, logs, scheduler, eval-session).
- **Phase 1 permission relay preserved** for the strict (non-`--dangerously`) path.
- **Test surface**: 304 tests across 33 files (up from 197 at v1.0). Zero TypeScript errors in `src/`.

## Architecture notes

- **VoxCPM2 + OpenAI = one provider factory.** VoxCPM2 served via `vllm serve --omni` speaks the same `/v1/audio/speech` shape as OpenAI; `makeHttpTTSProvider` covers both. Qwen (different shape) stays separate.
- **Triggers are Claude tasks, not shell commands.** Dangerous operations during trigger eval still go through Claude's own judgment (isolated eval uses `bypassPermissions` — matches `--dangerously` semantics; scheduler stall on permission prompts would defeat the purpose). Task content is user-owned.
- **Personas only affect proactive layer.** Reactive sessions use the base channel systemPrompt (plus a minimal appendix listing companion tools when enabled). Tonal shifts in user-initiated chat come from Claude's natural context-reading, not faux-identity roleplay.
- **Memory tier = markdown.** profile + personas + (future v1.2) recall + archive all live as hand-editable markdown. Grep-friendly, git-friendly, user-correctable.

## Known issues

- **WAV voice bubble vs file attachment**: VoxCPM2 outputs WAV. Current `buildMediaItemFromFile` classifies `.wav` as a generic file, so voice replies arrive in WeChat as file attachments rather than voice bubbles. Will be resolved in a follow-up (either transcode to MP3 or route through a dedicated voice item-type in ilink). Spike 4 results feed this.
- **Spike 4 (ilink voice format) + Spike 5 (eval latency baseline) + Spike 6 (tool-count impact)** not run. Plan Tasks 0-A/B/C remain deferred; v1.1 ships without these data points. If Spike 6 later shows tool-count degradation at 22 tools, v1.2 will split into MCP sub-servers (`wechat-core` / `wechat-voice` / `wechat-companion`).
- **No engagement-based push feedback** — v1.2 roadmap item.
- **No recall log** — v1.2. Profile is v1.1's only memory tier.
- **No archival summaries** — v1.3.
- **No custom persona creation tool** — users drop markdown files in `personas/` if they want a third persona; daemon picks them up.
- **No per-user admin gate** for `--dangerously` — convention-based (admin runs CLI, admin curates `access.json.allowFrom[]`). Code-enforced per-user gate is v1.2.

## Upgrade

```bash
cd ~/.claude/plugins/local/wechat
git pull
bun install  # adds croner
wechat-cc run              # strict mode (same behavior as v1.0)
# or:
wechat-cc run --dangerously  # personal single-admin mode
```

No state migration. No re-scan. Companion is opt-in — daemon behaves identically to v1.0 until you call `companion_enable` in WeChat.
