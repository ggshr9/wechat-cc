# v1.0.0-rc.1 (2026-04-22)

First Agent SDK rebuild candidate. See `docs/rfc/01-architecture.md` for the full architectural rationale.

## Highlights

- **New runtime**: MCP Channel plugin → single Bun daemon driving `@anthropic-ai/claude-agent-sdk@0.2.116`. Daemon owns one long-lived `query()` per project, streaming WeChat messages into an async prompt queue — Claude stays warm across turns.
- **Zero dialog on Windows**: dev-channel confirmation dialog is gone (Agent SDK headless path). Firewall popup from `share_page` is gone (Bun.serve now binds `127.0.0.1`). Workspace-trust didn't fire during Windows E2E either.
- **Per-project session pool**: up to 6 concurrent sessions, LRU evict at cap + 30 min idle sweep. Switching projects is instant (μs); the previous session stays resumable until evicted.
- **Permission relay via `canUseTool`**: Claude asks, daemon prompts on WeChat with a 5-char hash, user replies `y <hash>` or `n <hash>`. Timeout (10 min) auto-denies.
- **Default project fallback**: if you haven't run `/project add` + `/project switch` yet, messages route to whatever directory you launched the daemon from, aliased `_default`. Matches v0.x UX; eliminates the "silent drop on fresh install" hole.

## Breaking changes

- **`/restart` from WeChat is removed.** It was the root cause of the Windows death-loop (`.restart-flag` + dev-channel dialog combo). Use `/project switch` to change context, or `Ctrl+C` + `wechat-cc run` to restart the daemon process.
- **`wechat-cc run --fresh / --continue / --dangerously` flags ignored** (warning logged). Session lifecycle is internal to the daemon now.
- **`.mcp.json` at repo root deleted.** Claude Code's plugin system no longer auto-registers wechat as an MCP server. If you want the `/wechat:*` slash-command control plane (Phase 2), run `wechat-cc install --user`.
- **`.claude-plugin/plugin.json` deleted** — it only registered the MCP channel.

## State files preserved

No migration required. Files under `~/.claude/channels/wechat/` that the daemon still reads/writes (same formats as v0.x):

- `accounts/<id>/{account.json, token, sync_buf}`
- `projects.json`, `context_tokens.json`, `user_names.json`, `user_account_ids.json`
- `access.json`
- `docs/<slug>.md` + cloudflared binary under `bin/`
- `channel.log` (rotates at 10 MB)
- `inbox/` (incoming media, 30-day TTL)

Files that disappear: `.restart-flag`, `.restart-ack`, `current-cwd` — all were MCP-Channel-era hacks.

## Windows E2E verification

- ✅ Daemon started with no dialog (no dev-channel prompt, no workspace-trust, no Windows firewall popup)
- ✅ Two bound ilink accounts loaded, per-account long-poll started
- ✅ Inbound WeChat message → router (fallback to `_default` project) → Agent SDK session spawn → `reply` tool → WeChat received reply

## Internal

- Deleted: `server.ts` (2325 LOC), `start-channel.sh`, `.mcp.json`, `.claude-plugin/plugin.json`.
- Added: `src/core/` (session-manager, message-router, project-resolver, permission-relay, prompt-format), `src/features/tools.ts`, `src/daemon/` (main, bootstrap, single-instance, state-store, pending-permissions, media, poll-loop, ilink-glue).
- Removed top-level dep `@modelcontextprotocol/sdk`; added `@anthropic-ai/claude-agent-sdk`, `zod`, `@types/bun`. `typescript` moved to devDependencies.
- 98 unit tests across 13 test files in `src/`; full suite (including root-level legacy tests) at 194 passing, 0 failing.
- Zero TypeScript errors under `src/`. 10 pre-existing errors remain in root-level modules (`docs.ts`, `send-reply.ts`, `setup.ts`) — not in scope for Phase 1; will clean up opportunistically.

## Known

- `/wechat:*` Claude Code slash commands (plugin control plane) deferred to Phase 2.
- Agent SDK first-spawn overhead is ~11s per project (observed in Spike 1); subsequent turns are fast. Session pool keeps warm instances to amortize.
- `editMessage` tool sends a new message prefixed `(编辑后) ` — ilink has no true edit API.
- `util.test.ts > findOnPath('bun')` replaced with hermetic mocks — no more environmental flake.

