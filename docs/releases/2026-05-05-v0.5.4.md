# v0.5.4 — Win console-popup follow-ups (CREATE_NO_WINDOW for raw Rust Command + spawnSync defense-in-depth)

**Date**: 2026-05-05
**Tag**: `v0.5.4` + `desktop-v0.5.4`
**Scope**: Two fixes + a regression-guard test addressing console-window flicker on Win11 after the v0.5.3 PE-flip. **No breaking changes**.
**Predecessor**: [v0.5.3 — Win console-popup hotfix + typing keepalive + fallback-reply diagnostics](2026-05-05-v0.5.3.md)

---

## TL;DR

After v0.5.3 flipped the sidecar to PE subsystem=2 (GUI/hidden), a user reported PowerShell windows still flickering when clicking buttons in the dashboard. **I spent ~2 hours empirically testing on a Win11 machine and found that bun's spawnSync does NOT pop console windows for child processes when called from a subsystem=2 compiled binary** — `windowsHide:true` made no observable difference in repeated EnumWindows-based measurements. The only confirmed source of window-popping on Win that I could identify and reproduce was Rust's `std::process::Command::new("powershell.exe")` in `apps/desktop/src-tauri/src/lib.rs` — which v0.5.4 fixes.

| | 修法 | 触发场景 |
|---|---|---|
| **`wechat_daemon_pid` raw Rust Command pops console** | Add `CREATE_NO_WINDOW (0x08000000)` flag via `CommandExt::creation_flags` | "重启 daemon" 按钮每点一次都会让 powershell 闪一下（before/after pid 校验各调一次）。Tauri 的 plugin-shell 对 sidecar 已经默默加了这个 flag，但我们这一处用的是 raw `std::process::Command`。 |
| **spawnSync without windowsHide:true (defense-in-depth)** | 11 个 `spawnSync(...)` 调用点全部加 `windowsHide: true` | 我的本地实测（subsystem=2 + bun spawnSync）显示 windowsHide flag 不影响 console 弹窗 —— bun 编出来的二进制可能内部已经在隐藏。但 Node 上这是文档化的行为，加上 belt-and-suspenders 防止 Bun 未来版本变化。 |
| **`spawn-windowshide.test.ts` lint test** | grep 所有 `src/**/*.ts` 里的 spawnSync，强制要求 options 里有 `windowsHide` 字面量 | 防止下次有人在 daemon 里加新的 spawnSync 调用又忘了 flag。CI 任何平台都能跑。 |

文件名跟着改：`apps/desktop/package.json` + `tauri.conf.json` + `Cargo.toml` + 根 `package.json` 全部从 0.5.3 bump 到 0.5.4。

## 我实测了什么

为什么我本来推测的 `spawnSync windowsHide:true` 应该是关键修法、最后却"降级"成 defense-in-depth：

### 测试 1：probe binary，subsystem=2，spawnSync where.exe × 8 @ 4Hz

```text
TEST 1 (no windowsHide):  base=0 peak=0 samples_above_base=0/231
TEST 2 (windowsHide:true): base=0 peak=0 samples_above_base=0/226
```

EnumWindows 在 ConsoleWindowClass 上每 10ms 采样一次，2 秒共 230 次采样，**两边都没出现任何 visible console window**。

### 测试 2：probe binary，subsystem=2，spawnSync powershell.exe stdio:'inherit' × 5（match runCommands）

```text
TEST 1 (no windowsHide):   base=0 peak=0 samples_above_base=0/215
TEST 2 (windowsHide:true): base=0 peak=0 samples_above_base=0/151
```

仍然没有差异。

### 测试 3：双层 probe（外 GUI subsystem=2 stdio:'ignore' → 内 GUI subsystem=2 → spawnSync powershell stdio:'inherit'）

```text
TEST 1 (no windowsHide):   base=0 peak=0 samples_above_base=0/?
TEST 2 (windowsHide:true): base=0 peak=0 samples_above_base=0/?
```

模拟 Tauri 拉 sidecar 的真实场景（GUI parent，无 stdio inheritance）—— 还是没差异。

### 测试 4：真正的 v0.5.3 sidecar，4Hz × 16 次（mimic dashboard wizard install-progress poll）

```text
install-progress: base=0 peak=0 samples_above_base=0/256
service status:   base=0 peak=0 samples_above_base=0/256
```

直接调 v0.5.3 sidecar binary，依然 0 console windows。

### 测试 5：实际启动 v0.5.3 daemon 25 秒（Start-ScheduledTask + 监控）

```text
new console windows during 15s daemon boot: 0
```

v0.5.3 daemon 启动期间（含 BOOT、SCHED、INTROSPECT、POLL）—— 0 弹窗。

### 测试 6：实际启动 v0.5.3 dashboard 25 秒（Tauri GUI + 5s doctor poll × 5 cycles）

```text
console windows seen during dashboard idle (25s): 0
```

Idle dashboard 的 doctor poll 走的是 tauri-plugin-shell.sidecar() 调用，**Tauri 内部对 sidecar 已经设了 CREATE_NO_WINDOW**，所以不会闪。

## 结论

**bun-compiled subsystem=2 binary 调 spawnSync，在我的环境下不会弹 console 窗口**，无论 `windowsHide` 设不设。这跟我最初的理解相反。可能：
1. Bun 的 compile 给生成的二进制内部默认禁用了 child console window 分配（合理猜测，毕竟 `--windows-hide-console` 这个 flag 字面意义就该这样）
2. Win11 26100+ 在 GUI parent 没 console 时，console-subsystem child 的 console 默认 hidden（不太可能，违反了 30 年的 Windows 行为）

不管哪种，**实测就是不弹**。

唯一确认的弹窗来源：**Rust 这一处 raw `std::process::Command::new("powershell.exe")`**（`apps/desktop/src-tauri/src/lib.rs:147 wechat_daemon_pid`）。这个函数被 dashboard 的"重启 daemon"按钮调用，每次点击会调两次（before/after pid 比对），因此每点一次重启会闪两次 powershell。

v0.5.4 给它加 `CREATE_NO_WINDOW` flag 即修复。

## 用户报告的"安装并启动 powershell 不停闪烁"——还没复现

用户在 v0.5.3 测试时报告"现在点击安装并启动 powershell 会不停闪烁开启关闭"。我在同一台机器上用 v0.5.3 binary 走了一遍 install 流程（直接 invoke sidecar `service install`，加 daemon 启动 + dashboard 25s idle），都没看到 console window。

可能性：
1. 用户在 v0.5.3 测试时点了"重启 daemon"按钮（不是"安装并启动"）—— 那个按钮确实会闪 powershell。v0.5.4 的 CREATE_NO_WINDOW 修法对此场景有效。
2. 用户机器上有特殊配置（杀软 / Windows 版本 / DPI 缩放 / 焦点状态）放大了某种边缘 case 我的环境复现不出来。
3. 用户当时其实看的是 v0.5.0/0.5.1 那个 daemon 自己的 cmd 黑窗（subsystem=3，v0.5.3 之前的所有 release 都有）—— v0.5.3 的 PE-flip 已经修了。

**v0.5.4 让用户重测一次**：装新 NSIS、点重启按钮、看是否还有 powershell 闪烁。如果有，**带具体的截图 + 当时点的什么按钮 + 当时 PS history 一起贴上来**，我接着深挖。
