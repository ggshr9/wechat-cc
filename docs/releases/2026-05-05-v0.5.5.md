# v0.5.5 — compiled binary couldn't spawn its stdio MCP children

**Date**: 2026-05-05
**Tag**: `v0.5.5` + `desktop-v0.5.5`
**Scope**: 1 fix + 1 test + 4 small UX polish (dashboard cards no longer clip; sidebar version label de-staled; redundant solo-mode副文本去掉; conversations footer wording matches the dropdown reality). **No breaking changes.**
**Predecessor**: [v0.5.4 — Win console-popup follow-ups](2026-05-05-v0.5.4.md)

---

## TL;DR

The fix is one bug, but it's a load-bearing one for desktop-installer
users. Through v0.5.4 the compiled `wechat-cc-cli` binary **silently
shipped a broken stdio MCP spawn shape**: every `mcp__wechat__reply`
tool call vanished, the daemon fell into FALLBACK_REPLY for every turn,
the bot still replied (via the daemon's ilink direct fallback) but
sessions chat-view stopped pairing replies with inputs (renderer only
counts proper tool calls as outbound), and Claude itself reported
"wechat MCP server 这会断了" because from its end the tool surface
really was empty.

| | 修法 | 触发场景 |
|---|---|---|
| **MCP spawn shape broken in compiled binary** | `mcp-specs.ts` now picks the spawn args per `isCompiledBundle()`. Source mode keeps the script-path form. Compiled mode emits `args: ['mcp-server', '<name>']` — re-invoking the same binary with a hidden subcommand | Affects every desktop-installer install since RFC 03 / v2.0 introduced stdio MCP |
| **`mcp-server <wechat\|delegate>` subcommand** | `cli.ts` adds the hidden subcommand. Handler dynamic-imports the matching MCP entrypoint module, which reads its env vars + connects stdio at top level. The dynamic-import string is a literal so `bun build --compile` bundles both entrypoints (the previous cli.ts didn't import them anywhere, which is why the binary never shipped them in the first place) | Compiled-binary spawn target |
| **Test coverage** | `mcp-specs.test.ts` asserts both spawn shapes — source mode → script path, compiled mode → `['mcp-server', name]`. Test mocks `isCompiledBundle()` so it doesn't depend on which mode the test runner happens to be in | Regression guard |

Plus four small dashboard polish items shipped in the same release
(came out of using the v0.5.4 dashboard with real data on a 880-tall
window):

| | 修法 |
|---|---|
| **Conversations card clips its rows** | `.dash-body > .card { flex-shrink: 0 }`. The flex column was proportionally shrinking every card when their combined intrinsic height exceeded the viewport; combined with `.card { overflow: hidden }`, inner table rows were silently clipped. Repro: 4 accounts + 3 conversations + hero/update/config — conversations card showed "3 个会话" but rendered only 1.5 rows |
| **Sidebar version label `v1.2`** | Hardcoded since the v1.x → v0.x public-release reset. Now `v0.5.5` |
| **Redundant solo-mode副文本** | The dropdown text already says "/cc (Claude solo)"; the trailing "claude" mode-detail span only adds value for parallel/primary/chatroom modes. Now hidden in solo mode |
| **Conversations footer "只读视图"** | v0.5 introduced the inline mode-switch dropdown but the footer still said "在聊天里用 /cc /codex /both /chat 切换" as if the card was read-only. Now: "在这里直接切换，或在聊天里用 /cc /codex /both /chat。切换后会在该会话推送提示。" |

## 这个 bug 怎么没在 v2.0 被抓到

RFC 03 把 wechat tool 全部从 in-process MCP server 拆出去，搬到 stdio
MCP server。新的 spawn 流：daemon → SDK 拿 `mcpServers` config → SDK
spawn stdio child → child 走 internal-api 反向调 daemon。

`mcp-specs.ts` 用 `import.meta.url` 解析脚本路径——在源码模式下走真的
file:// URL，所以 `path.join(.., '..', '..', 'mcp-servers', name,
'main.ts')` 产出真实磁盘路径，spawn 起得来。所有 vitest / e2e / 开发
机本地测试都是源码模式，**所以这条路径在 CI 里看不到 bug**。

编译模式 (`bun build --compile`) 下 `import.meta.url` 解析到一条
`/$bunfs/...` 的虚拟路径。`path.join` 把它和 `..`/`..` 拼起来时把前缀
全吃掉了，最终产出 `/mcp-servers/<name>/main.ts`——一条根目录下不存在
的绝对路径。spawn 出来的 `wechat-cc-cli /mcp-servers/wechat/main.ts`
被 citty 当作未知子命令，输出 help text 后 exit 0。Claude SDK 看 stdio
里没有 MCP 协议字节，认为 server 不可用，**所有 `mcp__wechat__*` 工具
都不注册**。

第二个隐性问题：`cli.ts` **从来没 import 过 `src/mcp-servers/wechat/main.ts`
或 `delegate/main.ts`**。所以 `bun build --compile cli.ts` 的依赖图里
根本没这两个模块——它们从未被打进二进制。即便 spawn 路径正确，binary
里也没有 server 代码可执行。

修法的两层：
1. 让 `cli.ts` 通过 `mcp-server <name>` 子命令 dynamic-import 这两个模块（dynamic-import 字符串是字面量，bun --compile 能静态分析并打包）
2. 让 daemon 的 spawn 在编译模式下用 `args: ['mcp-server', '<name>']`，而不是脚本路径

源码模式行为不变。
