## CRITICAL hotfix details (PR #39)

### C1 ‚ÄĒ `SessionStore.deleteOne(alias, provider)` stops cross-provider clobber

`session-manager`'s stale-resume cleanup called `sessionStore.delete(alias)`, which removes **every** provider row for that alias. A chat using both `/cc` and `/codex` would lose **both** providers' resume points whenever one's row went stale (TTL past 7 days or jsonl missing). Result: the still-valid sibling provider was forced into a cold start (~10s) on every dispatch for the rest of the session.

Fix: added `deleteOne(alias, provider)` with a targeted `DELETE FROM sessions WHERE alias = ? AND provider = ?`. `session-manager:106` now calls `deleteOne(alias, providerId)`. The `/reset` admin command still uses the broad `delete(alias)` by design ‚ÄĒ its semantic is "forget this chat entirely". Regression test in `session-store.test.ts` + cross-provider integration test in `session-manager.test.ts` lock the new behavior.

### C2 ‚ÄĒ `send-reply.ts` ctx_token preflight + in-memory routing in daemon

The guard `if (!ctxToken && !persistedAccountId)` only rejected when **both** routing inputs were absent. When account routing existed (set on every inbound) but the context_token was missing or evicted, the code fell through to ilink with `context_token: undefined`, triggering ilink's 3√ó retry burst before failing ‚ÄĒ the user saw nothing and the logs blamed network errors.

Two parallel changes:
- **CLI fallback (`send-reply.ts`)**: split into `unknownChatIdError` (no routing at all) vs new `missingContextTokenError` (account known, session token gone). Reject at preflight on `!ctxToken` alone. Same fix in daemon's `assertChatRoutable` (was the same bug).
- **Daemon hot path (`ilink-glue.ts`)**: `adapter.sendMessage` previously routed through `sendReplyOnce`, which re-reads `context_tokens.json` from disk each call. `state-store` debounces disk writes 500ms, so a reply triggered immediately after the inbound that captured the ctx_token would see an empty file. Rewrote `adapter.sendMessage` to use the in-memory `ctxStore` and `acctStore` directly ‚ÄĒ no disk round-trip in the dispatch hot path. The CLI standalone (`wechat-cc reply`) still uses `sendReplyOnce` since it runs out-of-process. Broadcast paths (expired-notify, broadcast tool) keep using `sendReplyOnce` too ‚ÄĒ their targets have flushed ctx_tokens already.

E2E harness updated to mirror real ilink: every inbound now carries a `context_token: ctx-${chatId}` default. The old harness only forwarded it when explicitly set, which had silently worked because the fake-ilink-server doesn't reject missing tokens (production ilink does).

### C3 + C4 ‚ÄĒ log-viewer hostname binding + DOM-text escaping

`log-viewer.ts` ran two real attack surfaces:
- `Bun.serve` had no `hostname`, defaulting to 0.0.0.0. `channel.log` contains WeChat message bodies, user IDs, and bot-token fragments; any host on the LAN could poll `/api/logs` unauthenticated. Pin to `127.0.0.1` to match `docs.ts:632`.
- The client rendered log lines with `div.innerHTML = formatLine(line)`. Log lines originate from inbound WeChat messages (attacker-controllable); a crafted message body could inject HTML or JS into the operator's browser. Replaced the regex-into-HTML-string approach with a token-based `renderLine()` that builds a DocumentFragment via `createElement` + `textContent`. Every variable segment now goes through `.textContent`.

Smoke-tested: localhost connects (HTTP 200), external interface refused (connection reset).

### C5 + C6 ‚ÄĒ desktop CSP + DOM-constructed lightbox img

Two defense-in-depth fixes for the Tauri webview:
- `tauri.conf.json` had `csp: null` ‚ÄĒ webview had **no** Content Security Policy. Any XSS path could call `window.__TAURI__.core.invoke` and reach the OS via Tauri commands. Set a minimal CSP: `default-src 'self'`, `script-src 'self'`, `img-src 'self' data: blob: asset: http://asset.localhost https://asset.localhost`, `connect-src 'self' ipc: http://ipc.localhost https://ipc.localhost`, with `'unsafe-inline'` retained for styles pending tightening.
- `main.js openImageLightbox` interpolated a browser-resolved attachment URL into an attribute context via innerHTML ‚ÄĒ `src="${src}"` is vulnerable to `"` escapes in a crafted ilink payload, bypassing CSP's `img-src` whitelist. Rewrote to `createElement('img')` + `.src` setter ‚ÄĒ never parsed as HTML.

Verified via the new playwright `csp.spec.ts` (PR #40, separate test-infra branch): 18/18 playwright specs pass with the production CSP injected as a `<meta http-equiv>` tag into the served `index.html`. Sanity-checked by replacing the CSP with `img-src 'none'` and confirming the lightbox spec fails (exit 1) ‚ÄĒ the detector is not a no-op.

### Claude auth env bridging + Codex probe harden (PR #35)

The daemon spawns `claude` as a long-running subprocess at startup. When the user runs `claude login` interactively, it rotates credentials in the keychain, but the daemon's subprocess inherits the env from the original boot ‚ÄĒ it can't see the new credentials. After enough time, the subprocess emits the binary's "Please run /login" sentinel as assistant content (the v0.5.17 issue this builds on).

PR #35 bridges the auth env across the subprocess boundary by re-reading the user's `~/.config/claude/auth.json` (or platform equivalent) and re-injecting on each dispatch. Codex CLI version probe was strengthened to refuse provider registration on `codex --version` ‚Č† bundled `@openai/codex` version mismatch with a loud `[BOOT]` log.
