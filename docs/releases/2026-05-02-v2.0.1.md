# v2.0.1 — State → SQLite + CLI → citty (2026-05-02)

主题：**架构清理两件事一起 ship**：daemon 状态从 7 个 JSON/JSONL 文件搬到一个 SQLite 库,CLI 解析器从手写 switch 全搬到 [citty](https://github.com/unjs/citty)。**纯重构,零功能变化**,但因为涉及一次性数据迁移,值得单独发个 patch。

## TL;DR

- **Daemon 状态**:`session-state.json` / `sessions.json` / `conversations.json` + 4 个 per-chat `.jsonl`(events / observations / milestones / activity)合并到 `~/.claude/channels/wechat/wechat-cc.db`(`bun:sqlite`,无新增依赖)。
- **首次启动自动迁移**:升级后第一次 `wechat-cc run`,daemon 把现存 JSON/JSONL 读进 SQLite,然后把老文件 rename 成 `<原名>.migrated`。**保留一个版本作为保险期**,v2.0.2/v2.1 才会真删。
- **CLI 解析器**:24 个 subcommand 全部从手写 `switch(parsed.cmd)` 搬到 `defineCommand` + `subCommands`。每个 subcommand `--help` 现在自动生成,带 valueHint + description。
- **没动**:你的 chat 记录 / `accounts/` 凭证 / `memory/*.md` / `inbox/` 媒体 / `channel.log` 全部不动,跟之前一样。

## Highlights

### 1 · State → SQLite

7 张表(`session_state` / `sessions` / `conversations` / `activity` / `milestones` / `observations` / `events`),`PRAGMA user_version` 做 schema 版本号(现在 = 7),WAL 模式 + 5s busy_timeout。daemon 是单一写者,`wechat-cc sessions list-projects` 等 CLI 子命令并发读不会相互阻塞。

设计决策:
- **不引入 ORM**。直接 `db.prepare(...).all()/.get()/.run()`。bun:sqlite 跟 better-sqlite3 API 兼容,要换走也是局部改动。
- **STRICT 表 + 检查约束**。`conversations.mode_kind` 用 `CHECK (mode_kind IN ('solo', 'primary_tool', 'parallel', 'chatroom'))` 在 db 层挡掉拼错的 mode。
- **复合 PK 修了 v0.x 一个隐 bug**:legacy `sessions.json` 一个 alias 只能存一个 `session_id`,如果你 `/cc` 后再 `/codex`,Codex 的 session_id 会覆盖 Claude 的,切回 `/cc` 就没法 resume 了。新表 PK = `(alias, provider)`,两个 provider 的 session_id 各存各的,`/cc ↔ /codex` 来回切都能 warm resume。

### 2 · CLI → citty

24 个 subcommand 一次性清空老 switch:
- 单层:`status` / `list` / `install` / `doctor` / `setup-status` / `logs` / `run` / `setup` / `setup-poll` / `service` / `reply` / `update`
- 嵌套:`events list` / `observations list|archive` / `milestones list` / `conversations list` / `sessions list-projects|read-jsonl|delete|search` / `avatar info|set|remove` / `guard status|enable|disable` / `provider show|set` / `memory list|read|write` / `account remove` / `daemon kill` / `demo seed|unseed`

所有 `--help` 现在自动生成。`wechat-cc service --help` / `wechat-cc events list --help` 之前都是没有的。

裸 `wechat-cc` / `wechat-cc help` / `wechat-cc --help` 仍然走手写的 `HELP_TEXT`(里面有 RFC 链接 / 升级注意 / `--dangerously` 详细说明,citty auto-help 不会有这些)。

## ⚠️ 升级注意

### 第一次启动会发生什么

```
~/.claude/channels/wechat/
├── wechat-cc.db                     ← 新增,所有状态都在这里
├── wechat-cc.db-wal                 ← 新增,SQLite WAL 文件,正常
├── wechat-cc.db-shm                 ← 新增,SQLite 共享内存,正常
├── session-state.json.migrated      ← 老文件,已搬走,留作保险
├── sessions.json.migrated           ← 同上
├── conversations.json.migrated      ← 同上
└── memory/
    └── <chat_id>/
        ├── activity.jsonl.migrated      ← 同上
        ├── events.jsonl.migrated        ← 同上
        ├── milestones.jsonl.migrated    ← 同上
        ├── observations.jsonl.migrated  ← 同上
        ├── *.md                         ← 不动
        └── _handoff.md                  ← 不动
```

如果你看到 `*.migrated` 文件**别删**——它们是回滚保险,v2.0.2/v2.1 的某次升级会清理。

### 如果迁移失败怎么办

迁移路径每个 store 是独立的 + 幂等的。如果某个文件读出来 JSON 格式错误,daemon 不会 rename 它(保留原状)→ 下次启动重试。如果 INSERT 阶段崩了(磁盘满 / 权限错),老文件原状保留,db 里可能有部分行,但下次启动会被 INSERT OR REPLACE/IGNORE 收拾干净。

最坏情况下需要回滚:

```bash
cd ~/.claude/channels/wechat
rm wechat-cc.db wechat-cc.db-wal wechat-cc.db-shm
for f in *.migrated; do mv "$f" "${f%.migrated}"; done
find memory -name '*.migrated' | while read f; do mv "$f" "${f%.migrated}"; done
# 然后降回 v2.0.0:git checkout v2.0.0 && bun install
```

### Daemon × CLI 并发写

WAL 让并发读永不阻塞,但**两个进程同时写**(daemon 处理 inbound + 你在另一个终端跑 `wechat-cc sessions delete`)会触发 SQLite 的 busy 重试。我们把 `busy_timeout = 5000ms`,正常情况下察觉不到。如果你看到 `SQLITE_BUSY` 报错,说明你卡了一个 5 秒以上的写事务——大概率是 bug,issue 报上来。

### 跑测试需要 Bun

`package.json` 的 `test` script 从 `vitest run`(默认 Node)改成了 `bun --bun vitest run`。`bun:sqlite` 是 Bun 内置,Node 加载不了。CI 同步改了。如果你本地有 IDE 集成在跑 vitest,需要确保它走 Bun 运行时(VSCode vitest 插件:`vitest.commandLine: "bun --bun vitest"`)。

## Tech notes

| | v2.0.0 | v2.0.1 |
|---|---|---|
| State 文件数 | 3 顶层 JSON + 4 jsonl × N chats | 1 SQLite 文件 |
| State 总写次数 / 入站 | 3-5 fs writes(append + rewrite + rename) | 1 transactional INSERT |
| `archive(observation_id)` 写 I/O | O(N):读全文 + 改 + rewrite | O(1):单 UPDATE |
| `milestone fire` dedup | read-then-write race | INSERT…ON CONFLICT DO NOTHING |
| CLI 解析器 | 手写 `switch(cmd)`,~200 行 | citty `defineCommand`,自动 `--help` |
| `--help` 覆盖 | 仅顶层 HELP_TEXT | 顶层 + 每个 subcommand + 每个 nested verb |
| typecheck / depcheck | clean | clean |
| 测试数 | 1055 | 1083 |
| 新增运行时依赖 | — | — (`bun:sqlite` 是 Bun 内置;`citty` 是 dev-only) |
