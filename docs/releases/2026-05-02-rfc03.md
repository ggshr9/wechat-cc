# v2.0.0 — RFC 03 Multi-agent (2026-05-02)

主题：**让 Claude 和 Codex 在一个微信会话里同时上桌**。

7 天完成（RFC → P0 → P5 → review C1-C4 → P5.2 dashboard），含一次完整架构 review + 12 个 finding 全修。

## TL;DR

- **四种 mode**，按需在聊天里直接切：`/cc`（仅 Claude）/ `/codex`（仅 Codex）/ `/both`（两个并行答）/ `/chat`（圆桌互相讨论）。
- **架构开放**：不是 Claude+Codex 二元，而是 `ProviderRegistry` + `ProviderId = string`。Cursor / Gemini / 任意 SDK 都能加进来——见 RFC 03 Appendix D。
- **Auth-agnostic**：Codex 订阅或 API key 走同一条路，daemon 不知道也不关心。
- **MCP 去耦**：21 个 in-process tool 全部迁到 stdio MCP server，通过 daemon 内部 HTTP API 反向调用。能给任意 SDK provider 重用。
- **零数据迁移**：现有用户升级即用，未设 mode 的 chat 默认 solo。

## Highlights

### 1 · `/cc` `/codex` `/both` `/chat` — 四种对话模式

| Mode | 触发 | 谁说话 | 用户看到什么 |
|---|---|---|---|
| **Solo** | `/cc` 或 `/codex` | 一个 provider | 单条回复 |
| **Parallel** | `/both` | Claude + Codex 各答一遍 | `[Claude] ...` / `[Codex] ...` 两条 |
| **Primary + Tool** | `/cc + codex` | Claude 主答，Codex 当工具 | 单条回复，Claude 自决何时通过 `delegate_codex` 借力 |
| **Chatroom** | `/chat` | 两个互相讨论再答你 | `@user` 给你 / `@codex` 给同伴；多轮自然终止或 `max_rounds` 强制结束 |

每个 chat 自己持久化 mode（`conversations.json`），切完 daemon 重启不丢。

### 2 · 圆桌（`/chat`）的 `@-tag` 协议

Chatroom 模式下 assistant 输出按行首 `@-tag` 切片：
- `@user 我们一致认为是 X` → 给用户看（`[Claude]`/`[Codex]` 前缀）
- `@codex check src/foo.ts:42` → 给同伴看（用户也看见，但视为内部讨论）
- 不带前缀的段视为 `@user`
- `max_rounds=4` 安全上限。耗尽时附 "已达到 max_rounds" 后缀

`/stop` 立即取消正在进行的圆桌 loop（AbortController）。

### 3 · Dashboard 只读模式展示

桌面 GUI 新增"会话模式"卡片：每个 chat 一行，`Solo / Parallel / Primary / Chatroom` 四种 tint pill + 参与者副文本。10s 周期 poll，flip 由聊天里的 slash 命令完成（GUI 只展示，不操作——保持单一控制面）。

### 4 · MCP 去耦：所有工具走 stdio + 内部 HTTP

v1.x 的 in-process MCP server (`src/features/tools.ts`) 被删除，替换为：

- **`src/mcp-servers/wechat/main.ts`** — stdio MCP server，由 SDK 自动 spawn
- **`src/mcp-servers/delegate/main.ts`** — `delegate_<peer>` 工具，让 primary session 调远端 SDK
- **`src/daemon/internal-api.ts`** — daemon 暴露的 localhost-only HTTP，bearer token + `crypto.timingSafeEqual` + 0o600 token 文件 + 递归深度头部

效果：相同 21 个工具（reply / share_page / memory_* / companion_* / projects / voice / broadcast / set_user_name / resurface_page），任何遵循 MCP stdio 协议的 SDK 都能用。

### 5 · Codex SDK provider

`src/core/codex-agent-provider.ts` — 新增对等 provider：
- 持久化 `threadId`（跨 daemon 重启 resume）
- 工具调用走同一组 stdio MCP server（无 in-process fallback）
- `appendInstructions` 注入到首次 dispatch（Codex SDK 无 system_prompt 字段的妥协）
- 订阅 / API key 透明：不显式传 `apiKey`，让 `codex login` 或 env var 自然生效

### 6 · 架构开放

- `ProviderId = string`（开放 brand），不是 enum
- `ProviderRegistry` 统一注册 + lookup
- `prompt-builder.ts` mode-aware 段落组装，加新 mode 改一处
- `modeRequiresParticipantPrefix(mode)` 单一来源——加 N-way 圆桌只动一个函数
- 全部决策见 RFC 03 §3 + Appendix D

## 你看不到但很重要的修

review 后 12 个 finding 全部修复，分 4 个 commit：
- **C1** — chatroom 双发送守护 + mode-aware system prompt（删了之前 inline `buildChannelSystemPrompt` 的硬编码）
- **C2** — chatroom abort + `lastChatroomSpeaker` GC，避免 `/chat` 切走后状态残留
- **C3** — mode helper 抽取 + delegate hardening（depth>0 拒绝 / cwd 必须绝对路径 / 无 internalApi 时不挂 stdio server，避免泄漏）
- **C4** — chatroom envelope 压缩（round1 满协议 ~250 字符，round2+ 简记 ~80 字符；4 轮 loop 累积省 >300 字符）+ Codex resume + bootstrap 覆盖测试

## Breaking changes

- **删除 `src/features/tools.ts`**（in-process MCP server）。任何外部代码 import 它会报错——但这是私有模块，repo 外不该有依赖。
- **`canUseTool` 现在 callback 仍走 daemon**，但工具列表来源换成 stdio MCP——permission gating 协议未变。
- **`bootstrap.sdkOptionsForProject` 签名扩展**：必须传 `internalApi: { baseUrl, tokenFilePath }`，否则不会挂载任何 stdio server（fail-safe，不漏请求）。

## Internals

- **`src/core/conversation.ts`** — Mode discriminated union + ProviderId brand
- **`src/core/conversation-coordinator.ts`** — 中心 dispatch；per-chat `inFlightAborters` + `cancel(chatId)` 支持 `/stop`
- **`src/core/chatroom-protocol.ts`** — `parseAddressing()` 行首正则切片 + `wrapChatroomTurn()` 封套生成
- **`src/core/prompt-builder.ts`** — 替代 inline 字符串拼接，按 `delegateAvailable / companionEnabled / multiMode` 组装段落
- **`src/daemon/mode-commands.ts`** — slash 命令解析 + 帮助文本
- **`src/mcp-servers/wechat/`** — stdio server (519 行 main.ts) + 客户端校验 (94 行 client.ts) + 集成测试 (569 行)
- **`src/mcp-servers/delegate/main.ts`** — bare delegate provider（无 MCP 加载，结构性防递归）+ depth header（防御纵深）

## Skipped / 推迟

- **`/triple` 三方圆桌**：架构上 `ProviderRegistry` 已 N-way ready，但工具协议（`@-tag` routing + max_rounds 经济性）需要单独 RFC。v2.1 候选。
- **GUI 内 mode 切换**：本 release 只读展示。设计上保持单一控制面（聊天里 slash），避免"两个地方都能改、不一致"的麻烦。如真需要再走 RFC。
- **persistent thread state per provider**：v2.0 只 Codex 有 threadId 持久化；Claude 走 SDK 原生 `resume`。N-provider 时统一抽象等真实需求出现。

## Known issues

- **Codex 的 `appendInstructions` 只在首次 dispatch 注入**：thread resume 后丢失（Codex SDK 限制）。影响：`/chat` 中 Codex 在 round 2+ 看不到 chatroom 协议——靠 envelope 内 `[chatroom round N/M]` 简记保护。
- **update.e2e.test.ts 6 个失败**：环境签名问题（git commit signing），repo 外 CI 通过。
- **`vi.fn` 类型推断在 mock.calls 上不友好**：测试里手动标注 `vi.fn(async (_a: T1, _b: T2) => {...})`。
