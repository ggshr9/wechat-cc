## TL;DR

v0.5.1 PR3 (#23) 加 `<AllowHardTerminate>true</AllowHardTerminate>` 到 task XML 时，`powershellInstallCommands()` 写成了 `New-ScheduledTaskSettingsSet -AllowHardTerminate $true`。**这个 cmdlet 上根本没有 `-AllowHardTerminate` 参数**（它暴露的反向开关叫 `-DisallowHardTerminate`，默认 `$false` 即默认就允许硬终止）；XML 元素名被当成了 cmdlet 参数名。

第一个 Win11 用户安装 → cmdlet 报"参数不存在" → `$settings` 为 null → `Register-ScheduledTask -Settings $null` 失败 → wizard 整个 (1/2) 步骤红色。

CI 三平台 build matrix 的 vitest 单测**只用字符串 grep** 检查生成出来的 PowerShell 脚本（`expect(decoded).toMatch(/AllowHardTerminate\s+\$true/)`）—— 字符串确实存在 → 全绿合并 → 真机炸。**这种"语法看着对，运行才知道不对"的 pattern 自 v0.4.0 至 v0.5.1 已经栽过 >10 次**（schtasks `/IT`、S4U、`Disable-ScheduledTask`、`AllowHardTerminate` …），每次都靠真用户报 issue 才发现。

## Validator 的工作原理

```ps
# 内嵌脚本（pseudo）：
$ast = [Parser]::ParseInput($userScript, [ref]$null, [ref]$errors)
$cmds = $ast.FindAll({ ... CommandAst ... }, $true)
foreach ($c in $cmds) {
  $cmd = Get-Command $c.GetCommandName()
  if ($cmd.CommandType -in 'Cmdlet','Function') {
    $valid = @($cmd.Parameters.Keys)
    foreach ($p in $usedParams) {
      if (($valid | Where-Object { $_ -like "$p*" }).Count -eq 0) {
        Write-Output "BIND: $cmdletName has no parameter -$p"; exit 3
      }
    }
  }
}
```

外部 `.exe` 调用（`Get-Command` 返回 `Application`）跳过 —— 它们的 flag 不能内省，强查会有 false positive。属性赋值（`$obj.Foo = 1`）不是 `CommandAst`，自然不会被检查；这正是 Tier 1 的修法可行的原因。

## 为啥不一并跑全 install + uninstall 真 ScheduledTask

考虑过：CI windows-latest 上 `wechat-cc service install` 真注册一个 `wechat-cc` task，跑 30 秒，再 `service uninstall` 收尾。否决因为：
- 注册需要 `New-ScheduledTaskPrincipal -UserId $env:USERNAME`，CI runner 用户名跟 fixtures 里 `'bob'` 不一致，要么动态读 username 让测试漂移、要么真用 `$env:USERNAME` 让 fixtures 里出现真实账号
- ScheduledTask 注册有副作用，cleanup 失败的话 runner 留着脏 task；后续 retry 看到旧 task 又走另外一条路径
- 真注册 + Start 后 daemon 会去找 ilink → 没账号 → 进 onboarding → kill 进程；过程不再是"测试 install 路径"，而是"测试 install + 整套 daemon"

`validatePowerShellScript()` 走纯静态 + Get-Command 内省的代价是约每脚本 1-2s 冷启动，3 个 plan 变体共 18 个 PS spawn ≈ 30s，windows-latest matrix 多吃 30s 是值的。Tier 3（声明式 XML）是更彻底的对齐，进入 v0.6.0 plan。

## Migration / 升级

老用户：
- Source-mode 用户：`git pull && wechat-cc service install` 即可
- Desktop binary 用户：等 `desktop-v0.5.2` 出，重新下载 bundle；本次只换 sidecar binary，桌面 wizard / dashboard 行为不变

如果你已经在 v0.5.1 装失败留下了一个 broken state：
```ps
Get-ScheduledTask wechat-cc -ErrorAction SilentlyContinue | Unregister-ScheduledTask -Confirm:$false
```
然后跑 v0.5.2 的 install。
