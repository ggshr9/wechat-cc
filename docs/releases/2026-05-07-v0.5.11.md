# v0.5.11 — sessions detail no longer breaks for codex sessions

**Date**: 2026-05-07
**Tag**: `v0.5.11` + `desktop-v0.5.11`
**Scope**: 2 small fixes in `sessions read-jsonl` + 2 new playwright regression tests. **No breaking changes.**
**Predecessor**: [v0.5.10 — chatroom round-1 must-continue](2026-05-06-v0.5.10.md)

## What broke

User dogfooding the dashboard's 会话 tab after a stretch of `/codex` and `/chat` testing:

1. Click into the `compass` project → "**读取失败：ENOENT: no such file or directory, open '/tmp/wechat-cc-shim-...json'**". Cryptic.

Root-cause was a 2-bug stack:

| Bug | Symptom | Why |
|---|---|---|
| `sessions read-jsonl --json --out-file <tmp>` ignored `--out-file` on error paths | Shim's via-file route reads the never-written tmp file and gets ENOENT | The "no such alias" + "jsonl missing" branches used `console.log(JSON.stringify(...))` directly instead of `emitJson(..., outFile)`. Stdout-pipe path filled, file path stayed empty |
| Most-recent provider for `compass` was codex (after `/chat` testing), but `read-jsonl` hard-codes claude's jsonl location | Even if --out-file *had* worked, the path resolves to `~/.claude/projects/.../<codex-thread-id>.jsonl` which obviously doesn't exist. User saw "jsonl missing" in older deployments; ENOENT in this one | Codex jsonls live under `~/.codex/sessions/<YYYY>/<MM>/<DD>/rollout-<TS>-<thread>.jsonl` with a totally different schema. Reading them is a v0.6 task |

## Fix

`cli.ts sessions read-jsonl`:

```diff
- console.log(args.json ? JSON.stringify({ ok: false, error: 'no such alias' }, null, 2) : 'no such alias')
+ if (args.json) emitJson({ ok: false, error: 'no such alias' }, outFile)
+ else console.log('no such alias')
```

```diff
- console.log(args.json ? JSON.stringify({ ok: false, error: 'jsonl missing' }, null, 2) : 'jsonl missing')
+ if (args.json) emitJson({ ok: false, error: 'jsonl missing', path }, outFile)
+ else console.log('jsonl missing')
```

Plus a new branch that detects codex sessions cleanly:

```ts
if (rec.provider && rec.provider !== 'claude') {
  if (args.json) emitJson({
    ok: false,
    error: `session was last touched by ${rec.provider}; jsonl viewer only supports claude sessions today`,
    provider: rec.provider,
  }, outFile)
  ...
}
```

The dashboard now renders **"session was last touched by codex; jsonl viewer only supports claude sessions today"** in the empty-state slot — clear about why and what's missing.

## Playwright regression tests

`apps/desktop/playwright/sessions.spec.ts` (new):

1. **Shim API test**: invoke `wechat_cli_json_via_file` with a non-existent alias. Asserts the shim returns a clean envelope (`{ ok: false, error: 'no such alias' }`) instead of an ENOENT throw. This is the direct "did `--out-file` fire on error path" assertion.
2. **End-to-end test**: boot dashboard, force `data-mode=dashboard`, drive the same invoke through `window.__TAURI__.core.invoke`. Asserts the result envelope is intact end-to-end. Catches future regressions where either the shim, the polyfill, or the CLI's emitJson logic regresses.

Both pass under the existing playwright runner. Total wall-clock: ~2s.

## Out of scope

Codex jsonl viewing in the dashboard. The codex session format is different from claude's (different turn shape, different file layout), and the existing turn-rendering code in `apps/desktop/src/modules/sessions.js` is claude-specific. Wiring codex up is a v0.6 sessions-pane rewrite, not a patch.
