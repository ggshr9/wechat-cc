Hardens the daemon against the five reported hang/zombie/lock/model bugs, adds a structured per-turn observability trail, and ships an admin-only **self-diagnosis & self-healing** surface — the operator can ask the bot, in plain language, to check why a chat stopped replying and fix it. ## TL;DR

1. **Daemon never wedges on a stalled turn.** A per-turn watchdog bounds every dispatch (solo / `/both` / `/chat`): a silently-stalled agent turn is abandoned, its session released, and the user told to retry — instead of the chat going dark forever. Orphaned agent subprocesses are now reaped, and a half-started/wedged daemon no longer holds the single-instance lock hostage.
2. **`/model` takes effect without a restart.** Switching the pinned Claude model now applies to the next session per chat — no daemon restart.
3. **Per-turn observability trail.** Every turn's outcome (`completed` / `timeout` / `auth_failed` / `error`), duration, and reply status is persisted (SQLite, survives restart) and queryable — across solo, parallel, and chatroom modes.
4. **AI-native self-healing (admin-only).** Ask the bot "why did this chat stop replying — check and fix it" and the agent can inspect turn outcomes / live sessions / daemon health, release a wedged session, switch the model, or restart the daemon — each with a verification read-back, all gated to the admin tier.
5. **乙 v2 — WebSocket task channel.** Hands can now connect outbound over WS (handshake + reconnect) in addition to push transport.

## Daemon stability (the five reported bugs)

All TDD-driven; the suite stayed green throughout (2529 passed / 8 skipped).

- **Per-turn watchdog + self-heal.** `collectTurn` gained an idle timeout (default 10 min, `WECHAT_TURN_TIMEOUT_MS`). When an agent turn goes silent past the window, the turn is abandoned, the (now-poisoned) session is released so the **next** message spawns a fresh subprocess, and the user gets a "处理超时了，请稍后重发" notice. Wired across all three dispatch modes — solo `return`s, parallel `continue`s (the other provider's reply still goes out), chatroom `break`s.
- **Zombie subprocess reaping.** The Claude SDK `query()` now runs under an `AbortController` that `close()` trips — the documented "stop and clean up resources" teardown — so the claude child is reaped even when `interrupt()`/`close()` no-op against a dead transport.
- **Health-aware instance lock.** A `server.heartbeat` is stamped on every successful poll cycle (and after each delivered message). A holder whose heartbeat goes stale (it's wedged or never started serving) is treated as a stealable lock instead of blocking a fresh daemon with "another daemon already running". A missing heartbeat reads as fresh — an unproven lock is never stolen. The stale window is sized to exceed one max-length turn so a long-but-legitimate turn never trips a false steal.
- **`/model` lazy re-read.** The pinned Claude model is re-read per spawn via an mtime-cached config reader (one `stat`, parse only on change) instead of being captured once at boot.
- **launchd `ThrottleInterval`.** The macOS plist now caps KeepAlive respawn frequency (10 s), matching systemd's `RestartSec`, so a daemon that crashes on boot can't respawn in a tight loop.

## Observability — the per-turn trail

- Every dispatched turn emits a structured **TurnRecord** (chat / provider / mode / outcome / duration / reply-called / text-chunk count / error). Persisted to a new `turn_records` SQLite table (migration v15), pruned per-chat, so a hang or crash is diagnosable *post-mortem*.
- Surfaced on the daemon's internal API: `GET /v1/turns?chatId=&limit=` (a chat's recent turns, or recent across all chats), with `GET /v1/sessions` and an enriched `GET /v1/health` (turn-store wired, live-session count, heartbeat freshness).

## AI-native self-diagnosis & self-healing (admin-only)

The operator can drive daemon diagnosis and recovery from a WeChat conversation — no shell, no dashboard. Backed by admin-only MCP tools:

- **Diagnose (read-only):** turn outcomes for a chat ("why did it stop replying"), the live agent sessions (which is idle / wedged), and overall daemon health.
- **Remediate:** release a wedged session (self-heal — the next message gets a fresh subprocess), switch the pinned model, or restart the daemon. Each returns a verification read-back, and the mutating actions relay an "are you sure?" confirmation to the admin even at admin tier.

Gating is provider-agnostic: the daemon-control tools are registered for an admin session only, so a non-admin chat — on **either** provider — can't call or even discover them. (Codex has no per-tool callback, so this registration gate, not the per-tool permission relay, is what closes the door there.) New permission-tier kinds `daemon_introspect` (read) and `daemon_remediate` (mutating) are denied for trusted/guest.

**Known limitation:** the internal API authenticates a daemon-wide token, not a per-session identity. A shell-capable agent could read that token and call routes directly — a pre-existing property of *all* internal-api routes, not specific to these tools. Per-session token isolation is future work; until then, keep untrusted users out of `trusted`/`admin`.

## 乙 (one-brain-many-hands) v2 — WebSocket task channel

Hands can connect to the brain over an outbound WebSocket (authed handshake, reconnect, serve), routed by transport (`ws` → hub) alongside the existing push transport, with backward-compatible config. JSON-RPC 2.0 wire protocol; brain-side hub dispatch with response correlation and timeout.

## Other fixes

- **companion:** proactive pushes dispatch on the chat's mode provider; manual push trigger.
- **poll:** persist the ilink `sync_buf` cursor so a restart resumes from where it left off instead of replaying the unacked backlog (which re-ran the agent and re-sent rejected fallbacks).
- **model:** bump the Claude answer-model pin opus-4-7 → opus-4-8.
- **`/model` validation:** a bare alias or fast-mode tag (`opus`, `opus[1m]`) is now rejected instead of pinned and 404-ing every turn.
