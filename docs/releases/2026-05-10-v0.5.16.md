## What changed

### 1. Logs pipe truncation

- **`dab7e34` `fix(logs): route through --out-file to avoid bun-compile pipe truncation`**
  Reproduced the v0.5.14 sessions issue: bun-compiled CLI loses bytes when pushing 100s-of-KB JSON through a pipe (`pipe-buffer fills, EAGAIN, writes drop` вЂ” see `lib.rs:22-26`). At `--tail 200` with full path strings the logs payload crosses the threshold. CLI `logsCmd` now accepts `--out-file` and emits via `emitJson`; desktop `logs.js` switches to `wechat_cli_json_via_file`. Drop-in pattern, identical to sessions read-jsonl.

### 2. Sessions scroll snap

- **`e4f85ad` `fix(sessions): skip auto-refresh while user is actively scrolling`**
  Reproduced symptom: long conversation, scroll towards bottom, sometimes lands at the top instead. Root cause is `startDetailAutoRefresh` rebuilding `.phone-chat` via `innerHTML` every 4s вЂ” the new element starts at scrollTop=0, RAF restoration races layout. Smallest fix: re-use the existing `is-scrolling` class (added by `attachScrollbarFade` for ~700ms after the last scroll event) as a "user busy" guard. Skipped ticks pick up next cycle, so no new-message latency hit. Underlying layout race remains; if symptoms persist, follow-up is "Option B" (preserve `.phone-chat` element across refresh, swap inner content only).

### 3. Wizard rescan feedback

- **`7639224` `feat(setup): tell user what just happened on QR re-scan`**
  Same single message regardless of scan kind made small users re-scan repeatedly. Adds a `scenario` discriminator to the confirmed-status payload, computed from `accounts/` directory state just before `dedupeAccountsByUserId` runs:
  - `first` вЂ” no other active dirs
  - `reconnect` вЂ” same userId existed AND `sessionState.isExpired(oldBotId)` вЂ” fixing a known-broken connection
  - `redundant` вЂ” same userId existed AND still alive вЂ” re-scanned despite already connected
  - `new_account` вЂ” different userId existed вЂ” switched WeChat accounts
  
  Wizard renders scenario-specific copy via a `SCAN_SCENARIO_COPY` table; the small вњ“ badge gets a 2-char label per scenario (`е·Із»‘е®љ / е·Ій‡Ќиїћ / е·ІиїћжЋҐ / е·Іе€‡жЌў`). Spec at `docs/specs/2026-05-10-rescan-feedback.md`.

### 4. Session-expiry recovery

- **`3abf3a1` `fix(daemon): plug session-expiry recovery gaps reported by user`**
  Two gaps in the same recovery flow:
  - **`account remove` leaked SQLite `session_state` rows.** PR7 migrated the store from `session-state.json` to SQLite + renamed the file `.migrated`. The cleanup logic only knew about the legacy file в†’ every `account remove` on a previously-expired bot left an orphan row forever. Inject a `clearSessionStateBot` dep; CLI binds via `openWechatDb` + `makeSessionStateStore`; tests cover wired and unwired paths.
  - **Restart button only stopped launchctl/systemd-managed daemons.** A manual `wechat-cc run` started in a terminal holds the same `server.pid` lock but is invisible to launchctl, so the next `service start` would refuse with "another daemon already running", trigger a launchd `KeepAlive` loop, and silently fail. Add `wechat-cc daemon kill-residual` (reads `server.pid` в†’ `killDaemonByPid` with cmdline guard) and call it from dashboard's restart + stop buttons between the stop and start steps. Cross-platform вЂ” replaces the Windows-only `wechat_daemon_pid` Tauri shim.

- **`86fe066` `test(dedupe): document boot-time tied-mtime tiebreak risk`**
  Diagnostic-only вЂ” no code change. Boot-time dedupe runs without `keepBotId`, so it relies on filesystem mtime. On fast SSDs a quick remove + rescan can land both dirs within the same ms, and the lex tiebreak can archive the FRESHLY-bound dir if scan order doesn't match lex order. Setup-flow's own dedupe call passes `keepBotId` and is unaffected. Hardening (ctime, or `bound_at` field in `account.json`) is a v0.6-class follow-up; tests pin current behavior so we'll catch a regression if the rule changes.

## Migration

- **CLI users**: `wechat-cc update` picks this up like any other patch. The new `daemon kill-residual` subcommand is additive; no behavior change for users who don't restart via the dashboard.
- **Desktop users**: download new bundle from release page; `~/.claude/channels/wechat/` state untouched.
- **Anyone bitten by the session-expiry loop**: try (in order)
  1. `wechat-cc update`
  2. Click "й‡ЌеђЇ daemon" in the dashboard вЂ” the new residual-kill step frees the lock from any manual `wechat-cc run` left behind from previous troubleshooting.
  3. If the bot is still stuck on `errcode=-14`, `wechat-cc account remove <botId>` then re-scan. The SQLite `session_state` row will now actually clear; previously you'd accumulate orphan flags.
