## TL;DR

v1 cutover (RFC 04) post-mortem 暴露 3 个真实债 + 桌面 Playwright tier-2 长期 TODO。v0.5 一个 PR 把 4 块都做完：

- **A. Ref + wireRef helper** — 替代 `refs.x.current = lc` 裸写法；双重赋值抛错、未赋值 deref 抛错
- **B. wiring/ 5-file split** — `main-wiring.ts` 269 LOC → `wiring/{side-effects, tick-bodies, pipeline-deps, lifecycle-deps, index}.ts`，`main-wiring.ts` 整文件删除
- **C. daemon e2e 基础设施** — `fake-ilink-server` + `fake-sdk` + `harness` + `bootDaemon()` export + 独立 `vitest.e2e.config.ts` + 1 smoke test
- **D. 桌面 Playwright tier-2** — 7 specs (3 wizard + 2 dashboard + 2 interactions) + CI job

## 行为等价（硬约束）

`main.ts` 100 → 117 LOC 完全是结构化拆分：所有现有 daemon 行为、错误处理、shutdown 顺序不变。所有 1221 unit tests 跑过。`bootDaemon()` export 是新增非破坏。

## 架构改进 vs 之前

| 之前 | 现在 |
|---|---|
| `refs.polling.current = lc` 裸写法，无双重赋值检查 | `wireRef(refs.polling, lc)`，`Ref.set` 内置双重赋值抛 |
| `main-wiring.ts` 269 LOC，跨多职责单文件 | `wiring/` 5 文件按职责拆分，`index.ts` 纯 orchestration |
| daemon 不可程序化 boot/shutdown（仅 CLI 入口）| `bootDaemon(opts): DaemonHandle` 可独立 boot + 程序化 shutdown |
| 无 daemon e2e 基础设施 | fake-ilink-server / fake-sdk / harness 就位，配独立 vitest config + CI job |
| desktop UI 无交互测试 | Playwright 7 specs 跑 wizard / dashboard / interactions，配 CI job |

## v0.5 实施过程发现并修复的真 bug

| Bug | 修复 commit |
|---|---|
| `main.ts` 缺 `import.meta.main` guard，e2e harness 触发双重 boot + 锁冲突 | `import.meta.main` guard added |
| `fake-ilink-server` 用错 wire 字段（`updates` 实际应是 `msgs`、`sync_buf` 应是 `get_updates_buf`，per `transport.ts:74`）| 字段名修正 |
| `harness.stop()` 用 `process.kill(SIGTERM)` 会杀 vitest worker | `bootDaemon` 改返回 `DaemonHandle` 含 programmatic `shutdown()` |
| vitest 默认会捡到 `*.spec.ts` Playwright 文件 | `vitest.config.ts` exclude `**/playwright/**` |
| 根 tsconfig 没 `lib: dom`，Playwright spec 内 `document` 报 TS2584 | `tsconfig.json` exclude `apps/desktop/playwright/**` |
| P-T15 wiring 中发现：daemon 启动信息显示 `accounts=0` + `mode=strict` 即便用户传了 `--dangerously` | startup-sweeps 接受 `accountCount` + `dangerously` deps（v0.5 范围内修）|

## 推到 v0.6 的事

`src/daemon/__e2e__/README.md` 写了完整的 gap 分析。简版：

12 个 daemon e2e functional 测试（dispatch 4 modes、admin/mode/onboarding/permission/guard short-circuits、W-tier×2、signal-shutdown）需要：
- fake-ilink 加 `getconfig` + `upload` endpoints（typing 流程需要）
- fake-sdk → MCP tool bridging（reply tool 不真触发 ilink.sendMessage）
- harness 暴露 db handle（W-tier 写 SQLite，测试要查 db 而非 jsonl）
- 12 个测试文件

预计 ~4-6h 工作量。**当前 v0.5 ship 不依赖这部分** —— infra 已就位，下次 wiring bug 触发时按 README 步骤补即可（"motivated regression test"）。
