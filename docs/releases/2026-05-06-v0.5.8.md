## TL;DR

v0.5.6 made codex *findable*. Then real-world testing exposed three more cliffs sitting beneath it:

1. Codex's `findCodexPath()` resolves the platform binary via `moduleRequire.resolve(@openai/codex-linux-x64/...)`. The user's globally-installed codex CLI was 0.125.0, but our pinned SDK is 0.128.0 — wire-protocol drift meant codex emitted events the SDK couldn't parse, so dispatch returned empty and the user got no reply.
2. Even with version match, codex 0.128 silently auto-cancels every MCP tool call with `"user cancelled MCP tool call"` under `approval_policy="never"`, because MCP approval has its own gate that only the `dangerously_bypass_approvals_and_sandbox` config opens.
3. The dashboard's dev shim intercepted `mode set` updates into mock state but `conversations list` fell through to the real daemon — so dropdown clicks reverted on every 10s poll.
4. `/chat` mode (in-band `@-tag` protocol) was indistinguishable from `/both` in practice — both agents trained to "give the user a complete answer" rarely used the @-tags to actually discuss. Replaced with a moderator-driven architecture (AutoGen / CrewAI / orchestrator-worker pattern).

| | 修法 |
|---|---|
| **codex no reply (1+2)** | `findCodexBinary()` now probes `<wechat-cc>/node_modules/@openai/codex/bin/codex.js` first → guarantees SDK-version-matched binary. Codex provider sets `dangerously_bypass_approvals_and_sandbox=true` + `sandboxMode='danger-full-access'` whenever daemon runs `--dangerously` → MCP tool calls actually execute headlessly instead of getting cancelled |
| **dashboard dropdown revert (3)** | Shim now lazily seeds `__mockState.conversations` from the real CLI on first `conversations list`, then serves all subsequent reads from the mirror that `mode set` writes to. Dropdown writes and poller reads stay consistent |
| **/chat replaced with moderator (4)** | New `src/core/chatroom-moderator.ts` runs claude-haiku-4-5 each round to (a) pick which agent speaks next and (b) craft a targeted prompt that quotes the previous speaker's specific points. Hard-codes alternation. Forces final-round synthesis. ~3-5 haiku calls per /chat dispatch (~$0.01-0.05). Architecture matches AutoGen GroupChatManager / Anthropic orchestrator-worker. In-band `@-tag` protocol fully retired (no more `parseAddressing` / `wrapChatroomTurn` in the chatroom dispatch path) |
| **/chat conversations feel like real debate** | Moderator instructions emphasise "core goal: real交锋, not performance协作". Round 2 is a forced rebuttal turn ("must quote a specific claim and find ≥1 weak point — no '基本同意' allowed"). Final round emits a `[Codex · 终局]` prefix so users can tell synthesis from mid-debate |

## Bug-history side notes

The codex-no-reply path was a 3-layer onion:
1. **First layer (v0.5.5 fix)**: SDK couldn't reach platform binary at all — fixed by `codexPathOverride`.
2. **Second layer (v0.5.6 fix)**: `findCodexBinary()` walked PATH and found the right binary, but on this machine PATH had nvm's 0.125.0 which mismatched SDK 0.128.0 → empty events → no reply.
3. **Third layer (v0.5.8 fix)**: even with version-matched 0.128 binary, codex auto-cancels MCP tools without `dangerously_bypass_approvals_and_sandbox=true`. Discovered by `strings node_modules/@openai/codex/.../codex | grep dangerously_bypass`.

Each layer alone fixed nothing user-visible because the next layer also failed silently. Worth documenting because the same pattern is likely to recur with future Codex CLI version bumps.

## /chat architecture — why moderator over @-tag protocol

In-band protocols ask the speaker to self-route via @-tags. Both speakers' training priors push them toward "give the user a complete answer" — even with explicit instructions, the @-tag was rarely used in practice and `/chat` collapsed into `/both`. Mature multi-agent frameworks all defer routing to a separate coordinator/manager (AutoGen `GroupChatManager`, CrewAI hierarchical mode, OpenAI Swarm handoff tools, Anthropic orchestrator-worker). v0.5.8 follows the same pattern with claude-haiku-4-5 as the moderator.

Trade-offs:
- **+** Real back-and-forth instead of two parallel monologues
- **+** Targeted prompts that quote prior speakers' actual content (not "what do you think" generic)
- **+** Forced final-turn synthesis with `[· 终局]` visual marker
- **−** Cost: 3-5 extra haiku calls per /chat (~¥0.05–0.15 each)
- **−** Latency: +5-10s for moderator overhead
