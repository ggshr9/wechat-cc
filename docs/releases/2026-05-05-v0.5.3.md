## Bun upstream bug

`bun build --compile --windows-hide-console` 在 bun 1.3.13 上：

```text
$ bun build --help | grep windows-hide
      --windows-hide-console   When using --compile targeting Windows, prevent a Command prompt from opening alongside the executable
   ↑ documented and accepted

$ bun build --compile --windows-hide-console probe.ts -o probe.exe
   ↑ no error, exits 0

$ # Inspect resulting PE subsystem (offset peOff+0x5c)
$ # 2 = IMAGE_SUBSYSTEM_WINDOWS_GUI (no console allocated)
$ # 3 = IMAGE_SUBSYSTEM_WINDOWS_CUI (console allocated → cmd window pops up)
subsystem: 3   ← BUG: flag was no-op
```

我们的 workaround 在 `.github/workflows/desktop.yml` 里编完 sidecar 之后单独跑一步：

```pwsh
$bytes = [System.IO.File]::ReadAllBytes($path)
$peOff = [BitConverter]::ToInt32($bytes, 0x3c)
$before = [BitConverter]::ToUInt16($bytes, $peOff + 0x5c)
if ($before -ne 3) { Write-Error "..."; exit 1 }   # 万一 Bun 修了，让我们知道
$bytes[$peOff + 0x5c] = 2
$bytes[$peOff + 0x5d] = 0
[System.IO.File]::WriteAllBytes($path, $bytes)
# read-after-write verify, fail loudly if it didn't stick
```

**当 Bun 修复后**：CI 会在 build 时报错 `expected PE subsystem=3 (CONSOLE) before flip, got 2 — Bun may have fixed --windows-hide-console; remove this workaround`。看到那个就把这一步删了。

## Typing keepalive 设计

`mw-typing` 之前是 fire-once（在 inbound 到达时发一次 typing 然后调 next）。新版本：

```ts
let inflight = false
const tick = () => {
  if (inflight) return    // 上一次还没回来，跳过这次（避免堆积）
  inflight = true
  void deps.sendTyping(...).catch(() => {}).finally(() => { inflight = false })
}
tick()                             // t=0 立即发
const timer = setInterval(tick, 5_000)   // 之后每 5 秒一次
try { await next() } finally { clearInterval(timer) }
```

**5 秒**这个数字的依据：小于微信端 typing 提示寿命（5-10s）+ 远小于典型 NAT/server idle close（30-60s+）。tick 之间发出去的 typing 调用**和后续 sendmessage 共享 baseUrl + Bun 的 fetch keep-alive 池**——所以 typing 也是个 connection-pool warmer，sendmessage 上去时 pool 里至少一根连接是 5 秒之内验证过活的。从根上灭了 v0.5.2 release notes 里说的 "stale keep-alive 连接 → 30s timeout × 2 次 retry → 65 秒恢复" 这个失败模式。

## Fallback-reply 诊断重构

`bootstrap/index.ts:310` 之前是：

```ts
sendAssistantText: deps.ilink.sendMessage
  ? async (chatId, text) => { await deps.ilink.sendMessage(chatId, text) }
  : undefined
```

`await` 完丢掉返回值。`ilink.sendMessage` 的 type 是 `(chatId, text) => Promise<{ msgId, error? }>` —— **当 ilink 内部 retry 全失败时，返回 `{ msgId: 'err:...', error: '...' }`。这个 envelope 在 wrapper 里被无声吃掉，daemon 主流程没有任何 log，dashboard 看不到，只有 channel.log 深处有 `[RETRY_FAIL]`**。

v0.5.3 抽出来：

```ts
// src/daemon/bootstrap/fallback-reply.ts
export function makeSendAssistantText(deps: FallbackReplyDeps): SendAssistantText | undefined {
  if (!deps.sendMessage) return undefined
  return async (chatId, text) => {
    let result
    try { result = await deps.sendMessage!(chatId, text) }
    catch (err) {
      deps.log('FALLBACK_REPLY_FAIL', `chat=${chatId} threw: ${...}`)
      throw err
    }
    if (result.error) {
      deps.log('FALLBACK_REPLY_FAIL', `chat=${chatId} error=${result.error}`)
      return
    }
    deps.log('FALLBACK_REPLY_SENT', `chat=${chatId} msgId=${result.msgId}`)
  }
}
```

4 个单测覆盖：no-op when sendMessage undefined / log fail on error envelope / log success / log fail + re-throw on exception。
