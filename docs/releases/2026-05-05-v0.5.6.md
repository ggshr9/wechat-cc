# v0.5.6 — codex availability + wechat-userId-based dedupe + small UX

**Date**: 2026-05-05
**Tag**: `v0.5.6` + `desktop-v0.5.6`
**Scope**: 2 fixes + 2 UX polish + new `mcp-server` deploy verified end-to-end on the maintainer's running daemon. **No breaking changes.**
**Predecessor**: [v0.5.5 — MCP compiled-binary fix + dashboard polish](2026-05-05-v0.5.5.md)

---

## TL;DR

v0.5.5 fixed compiled-binary MCP. With MCP back, the `/codex` and dashboard-dropdown flows surfaced two more cliffs that were sitting under the FALLBACK_REPLY noise:

1. **Codex 实际不可用**: even with `@openai/codex-sdk` registered, the daemon-from-compiled-binary couldn't reach the platform binary at dispatch time, so every codex turn silently errored ("Unable to locate Codex CLI binaries"). Mode switch to `/codex` succeeded at the DB level → user sent `9-1?` → no reply.
2. **同一个微信 userId 出现两条 bot 行**: each QR scan creates a fresh `accounts/<botId>/` dir; ilink invalidates the old bot server-side but the local dir piles up. Result: dashboard shows two "顾时瑞" rows, the daemon polls a session-expired bot for hours, and the user's mental model ("ilink 只绑一个") is silently violated by the on-disk state.

Plus two related UX cleanups (one of which was a v0.5.5 carryover that's only worth shipping with a test-coverage update).

| | 修法 |
|---|---|
| **Codex 可用性** | New `findCodexBinary()` helper: walks `process.env.PATH` first, then `~/.nvm/versions/node/*/bin/` newest-first as a fallback for systemd-user-service environments that don't source NVM's PATH. Boot-time only (~1ms, free). When found, the codex provider is registered with the binary as `codexPathOverride`, bypassing the SDK's broken `findCodexPath()` resolution from inside the bun-compiled bundle. When not found, the codex provider is **not** registered — `validateMode('codex')` then 4xx's at the internal-api boundary, so the dashboard dropdown reverts visibly with an error border instead of silently swallowing dispatch errors per turn. |
| **Userid-keyed dedupe** | New `dedupeAccountsByUserId()` helper. Scans `accounts/`, groups by userId, archives older bot dirs to `<botId>.superseded.<iso>`. `loadAllAccounts` (daemon) + `readAccounts` (doctor) + `cli-status.printAccounts` (`wechat-cc list`) all skip the `.superseded.` infix — so the user sees one row per wechat user, while the audit trail stays on disk. Idempotent: running on already-clean state is a no-op. Wired into both the daemon boot path (handles legacy state on first v0.5.6 launch) and the setup-flow `persistConfirmedAccount` path (handles every fresh re-scan going forward, with an explicit `keepBotId` override so SSDs that haven't bumped mtime yet don't accidentally archive the just-written dir). |
| **Drop 配置摘要 (Configuration) card from dashboard** | UX audit: `Provider` was misleading once `/cc /codex /both /chat` made it per-chat, the binary path is already shown in the wizard's Agent step, allowlist count is in the bound-accounts card meta line, and state directory is a power-user concern (`wechat-cc doctor --json` exposes it). The whole card paid no rent. Removed the markup, the `configRows` export + its test, and the dashboard.js render call. |
| **Drop 账号 column from Conversations card** | Same audit, smaller blast radius: the user_id and account_id columns were showing two truncated 12-char hashes side-by-side. Functionally different but visually redundant in 1-user-1-bot setups; the bot routing is in 绑定账号 above. Now: 用户 / 模式. |

## Why not auto-`account remove` instead of archive?

Three reasons to rename rather than delete:
1. **Audit trail**: if the dedupe heuristic is ever wrong (a corner case I haven't thought of), the user can manually rename it back without re-doing the QR scan.
2. **Inflight conversations**: a session might still be mid-flight against the about-to-be-archived bot at the moment the daemon reboots; better to fail to forward replies than to delete the routing key.
3. **Accident recovery**: a sudo-driven `rm -rf accounts/<bot>/` from a panicked operator is unrecoverable; an `mv` to `.superseded` is reversible with a single `mv` back.

A future `wechat-cc account purge-superseded` (or equivalent) can do the actual deletion once enough time has passed (e.g., `--older-than 30d`).
