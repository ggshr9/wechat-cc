## What broke

User tested v0.5.9 and reported two regressions:

1. **Second user message in /chat got 0 replies.** Daemon log:
   ```
   [COORDINATOR] chatroom → start participants=claude,codex max=4 history=4
   [COORDINATOR_CHATROOM] round=1 moderator end (用户以网络梗式幽默回应，话题自然完成，无需续谈)
   [COORDINATOR] chatroom → done; history=4
   ```
   The neutralised moderator decided the user's casual follow-up didn't need a reply. **Zero speakers dispatched. User feels ignored.** This is correct from "no manufactured filler" perspective but wrong from UX — the user explicitly opted into /chat and is waiting for SOMETHING.

2. **Stale /chat confirmation message.** Switching to /chat still printed `'✅ 聊天室模式开启。Claude 和 Codex 会在你的下一条消息后轮流讨论（最多 4 轮内部 round），用 @ 路由（@user 给你，@codex / @claude 给对方）...'`. The "@ 路由" / "4 轮内部 round" framing is from the v0.5.7-era in-band protocol that v0.5.8/v0.5.9 retired.

## Fixes

| | 修法 |
|---|---|
| **Round 1 must continue** | When the moderator returns `action: 'end'` on round 1 of a new user msg, the evaluator coerces it to `action: 'continue'` with the generic prompt. Picks the speaker via forced alternation (peer of last-spoken). Subsequent rounds (2+) can still end normally. Implementation: ~10 LOC in `chatroom-moderator.ts`'s end-action branch. Logged as `[CHATROOM_MOD] moderator wanted end on round 1 ... coercing to continue` |
| **/chat confirmation rewritten** | New text: `"✅ 聊天室开启。Claude 和 Codex 都"在场"了——你之后每条消息他们都看得见对方的回复，会接着上下文讨论。每条回复带 [Claude] / [Codex] 前缀。切走（/cc /codex /solo）会清空聊天室上下文。"`. Removes obsolete @-tag / max-round mention. Reflects v0.5.9's persistent-session model |

## Tests

- New: `chatroom-moderator.test.ts` — explicit test that round-1 end is coerced to continue
- Updated: `conversation-coordinator.test.ts` — replaced "round 1 end → 0 dispatches" with "round 1 end → 1 coerced dispatch + round 2 end normally"
- Updated: `mode-commands.test.ts` — new `/chat` confirmation copy assertions

1307 passing + the same 2 pre-existing flakes. tsc clean.

## Lessons (note to self)

The moderator-as-arbiter design has an asymmetric failure mode: false-positive `end`s look like the bot ignored you, false-negative `end`s look like wasted rounds. The former is ~10× worse from UX, so the trade-off should always lean toward "say something". v0.5.10's "round 1 must continue" rule is a guardrail for the asymmetry; let the moderator be liberal with `end` on round 2+ where the user has at least one reply already.
