# Voice Spec (Phase 2 · Sub-spec 1)

**Status**: Draft · 2026-04-22
**Target**: v1.1
**Scope**: Outbound TTS only (inbound voice is already transcribed by WeChat and delivered as `[语音 <transcription>]` text)

---

## Goal

Add the ability for Claude to reply with audio via a new `reply_voice` tool, gated to contexts where the user explicitly asks for voice. Configuration happens inside WeChat (Claude walks the user through API-key registration and validation); zero terminal / env / file editing for the end user.

## What this ships

1. **One outbound tool**: `reply_voice({chat_id, text})` — renders `text` via TTS provider, uploads audio via the existing ilink `sendMessage` path with a voice `item_list` entry, returns `{ok, msg_id}`.
2. **Two configuration tools** callable by Claude during a chat:
   - `save_voice_config({provider, api_key, default_voice?, openai_api_key?})` — persists, tests, reports back.
   - `voice_config_status()` — returns `{configured: bool, provider?, default_voice?, has_openai_fallback: bool}` so Claude can check state without re-prompting.
3. **Provider adapter** in `src/daemon/tts/` for Qwen (`qwen3-tts-flash`, default) and OpenAI (`gpt-4o-mini-tts`, optional fallback). Interface `TTSProvider.synth(text, voice, language): Promise<{audio: Buffer; mimeType: string}>`.
4. **System-prompt delta**: a short rule teaching Claude when to use `reply_voice` vs. `reply`.
5. **Voice config file**: `~/.claude/channels/wechat/voice-config.json`, mode `0600`, shape documented below. Not shipped with the package.

## What this explicitly does NOT ship

- **Inbound STT** — WeChat does it. If WeChat failed the transcription (`[语音 · 未转文字]`, `[语音 · 空内容]`, `[语音 · 下载失败]`), Claude asks the user to retype/resend per the existing channel prompt rules.
- **Runtime `/voice on/off` toggle** — YAGNI. Tool-gating is the only trigger.
- **"Always voice" mode** — YAGNI. Cc-connect's `tts_mode: 'always'` is not replicated.
- **Symmetric voice-in→voice-out** — YAGNI. Claude may still choose `reply_voice` when user sent voice, but it's a Claude decision, not a daemon rule.
- **Local TTS (Kokoro/Piper/etc.)** — out of scope. Cloud-only.
- **Voice caching / re-use** — out of scope. Generate per reply.
- **Voice selection per message** — voice is a config preference, not a tool arg.

## Architecture

### Data flow (outbound)

```
WeChat user: "念一下 hello"
   │
   ▼ ilink → pollLoop → MessageRouter → SessionManager.dispatch()
   │
   ▼ Claude (Agent SDK headless) observes prompt
   │
   ▼ Claude decides to call reply_voice (per system-prompt rule)
   │
   ▼ tools.ts reply_voice handler:
      ├─ load voice-config.json
      ├─ TTSProvider.synth(text, voice, lang) → {audio, mimeType}
      ├─ save to temp file: tts-<timestamp>-<hash>.mp3
      ├─ buildMediaItemFromFile(...)  [existing in media.ts]
      ├─ ilinkSendMessage({... item_list: [voice_item]})
      └─ unlink temp file
   │
   ▼ returns {ok, msg_id} to Claude
```

### Configuration flow (first-time setup via conversation)

```
User: "念一下这段话"
   │
   ▼ Claude: reply_voice({...}) → {ok:false, reason:'not_configured'}
   │
   ▼ Claude: voice_config_status() → {configured:false}
   │
   ▼ Claude (via reply tool): "需要个 Qwen DashScope API key 才能发语音。
                               没有的话我教你注册；有了直接发过来。"
   │
   ▼ User: pastes "sk-abc..." OR says "教我"
   │
   ▼ (branch A) Key pasted:
   │    Claude: save_voice_config({provider:'qwen', api_key:'sk-abc...'})
   │    → tool writes voice-config.json (chmod 0600)
   │    → tool runs 1-second test synth ("测试" → audio)
   │    → returns {ok:true, tested_ms: 850}
   │    Claude via reply: "已验证。建议把刚才发 key 的消息删掉
   │                      （我也帮你把那条改成占位符）"
   │    Claude: edit_message({chat_id, msg_id, text:"[配置已保存]"})
   │
   ▼ (branch B) "教我":
        Claude uses its general knowledge / web tools to walk through
        https://dashscope.console.aliyun.com signup. No special tool.
```

Key properties:
- Configuration IS the first voice request, inline. No `wechat-cc setup-voice` command.
- API key lives in one file, chmod 0600, never in env or git.
- Tool tests the key before saving (1s test call). Saves user the "did it work?" confusion.
- `edit_message` overwrite of the key-bearing message is best-effort (not all ilink flows support edit; see §Phase 1 editMessage note — currently sends a new prefixed message).

### Error modes

| Failure | User sees | Code path |
|---|---|---|
| TTS provider 401 (bad key) | Claude text: "key 无效，重新发一次？" | `save_voice_config` test call detects; won't save |
| TTS provider 429/5xx (transient) | Claude text: "语音服务繁忙，先文字回复" | `reply_voice` returns `{ok:false, reason:'transient'}`; Claude retries with `reply` |
| ilink voice upload fails | Same as above | Caught inside `reply_voice` handler |
| `voice-config.json` missing / corrupt | `reply_voice` → `{ok:false, reason:'not_configured'}` → config flow | Detected at load time |
| Text exceeds 500 chars | `reply_voice` → `{ok:false, reason:'too_long', limit:500}` | Claude either shortens + retries or falls back to `reply` |
| Unknown voice name in config | Silently fall back to `Cherry` + log `[TTS] unknown voice 'X', using Cherry` | Don't error — user-friendly |

## Contracts

### `reply_voice` tool

```ts
tool(
  'reply_voice',
  '用语音回复用户。仅在用户明确要求语音回复时使用。文本不超过 500 字。不适合代码块、长 URL、或结构化列表。',
  { chat_id: z.string(), text: z.string() },
  async ({ chat_id, text }) => { /* handler */ }
)
```

Handler logic:
1. If `text.length > 500` → return `{ok:false, reason:'too_long', limit:500}`.
2. Load voice config; if missing/invalid → return `{ok:false, reason:'not_configured'}`.
3. `provider.synth(text, config.default_voice ?? 'Cherry', auto-detect)` → `{audio, mimeType}`.
4. Write audio to temp file under `<STATE_DIR>/tts-tmp/reply-<timestamp>-<pid>.<ext>`.
5. `buildMediaItemFromFile(tempPath, chat_id, baseUrl, token)` (existing infra).
6. `ilinkSendMessage(...)` with `item_list: [item]`. 3-retry wrapper handles transient CDN 5xx.
7. `unlinkSync(tempPath)` — best-effort, swallow errors.
8. Return `{ok:true, msg_id}`.

### `save_voice_config` tool

```ts
tool(
  'save_voice_config',
  '保存用户的 TTS 配置（API key、provider、默认音色）。调用后会立刻做一次测试合成验证。',
  {
    provider: z.enum(['qwen', 'openai']),
    api_key: z.string().min(10),
    default_voice: z.string().optional(),
    openai_api_key: z.string().optional(),  // for primary=qwen + optional OpenAI fallback
  },
  async ({ provider, api_key, default_voice, openai_api_key }) => { /* handler */ }
)
```

Handler logic:
1. Construct a provider instance from args; run a 1-second test synth on fixed text `"测试"`.
2. If test fails → return `{ok:false, reason, detail}` without touching disk.
3. Write JSON atomically (tmp + rename) to `voice-config.json`, chmod 0600.
4. Return `{ok:true, tested_ms, provider, default_voice: default_voice ?? 'Cherry'}`.

### `voice_config_status` tool

```ts
tool(
  'voice_config_status',
  '检查 TTS 配置状态。返回当前 provider / 音色 / 是否配置了 OpenAI fallback。',
  {},
  async () => { /* handler */ }
)
```

Return shape:
```ts
type VoiceConfigStatus =
  | { configured: false }
  | {
      configured: true
      provider: 'qwen' | 'openai'
      default_voice: string
      has_openai_fallback: boolean
      saved_at: string  // ISO
    }
```

Notably does NOT return the API key — Claude doesn't need it; reduces risk of leaking via conversation context.

### `voice-config.json` format

```json
{
  "provider": "qwen",
  "api_key": "sk-...",
  "default_voice": "Cherry",
  "openai_api_key": "sk-...",
  "saved_at": "2026-04-22T10:30:00Z"
}
```

Mode `0600` (owner read/write). Not backed up, not in git, never logged.

### `TTSProvider` interface

```ts
// src/daemon/tts/types.ts
export interface TTSProvider {
  readonly name: 'qwen' | 'openai'
  synth(text: string, voice: string, lang?: 'zh' | 'en' | 'auto'): Promise<{
    audio: Buffer
    mimeType: string  // 'audio/mpeg' for MP3, 'audio/ogg; codecs=opus' for OGG
  }>
  test(): Promise<{ok: true} | {ok: false, reason: string, detail?: string}>
}
```

Two concrete implementations:
- `src/daemon/tts/qwen.ts` — POSTs to `https://dashscope.aliyuncs.com/api/v1/services/audio/tts/` with `qwen3-tts-flash`, returns MP3.
- `src/daemon/tts/openai.ts` — POSTs to `https://api.openai.com/v1/audio/speech` with `gpt-4o-mini-tts`, returns MP3 (or configurable `response_format`).

Both return MP3 for consistency. Transcoding to whatever ilink wants is handled in the `reply_voice` handler, not the provider.

### System-prompt delta

Add to the existing channel instructions in `bootstrap.ts`:

```
- reply_voice 仅在用户明确要求语音回复时使用（消息中含 "语音回复" /
  "念一下" / "说给我听" / "speak it" / "say it aloud" 等）。其他情况
  一律用 reply 回复文本。reply_voice 不适合代码块、URL、长列表。
- 如果用户第一次要求语音但未配置，先调 voice_config_status 检查，
  未配置则调 reply 引导用户发 API key，再调 save_voice_config 保存。
```

## Spike 4 (Task 0 of impl plan)

**Question:** does ilink voice send API accept MP3 directly, or does it require OGG/Opus / AMR?

**Method:**
1. Render a 3-second "测试" via Qwen → MP3 buffer.
2. Use existing `buildMediaItemFromFile` + `ilinkSendMessage` with `UPLOAD_MEDIA_TYPE.VIDEO` (ilink's voice type id TBD — check `ilink.ts` for existing enum; add `VOICE: 6` or similar if absent).
3. Observe WeChat client behavior:
   - **PASS**: voice message renders, playable → spec proceeds as written.
   - **FAIL (wrong format)**: try sending MP3 as a file attachment. If ilink accepts, document as degraded UX ("voice arrives as a file, not a voice bubble").
   - **FAIL (wrong item type)**: inspect ilink SDK source or network traffic for the correct `item.type` id.

**Possible outcomes:**
- (best) ilink accepts MP3 as-is → no transcode, ship simple.
- (meh) ilink requires OGG/Opus → add FFmpeg dep (`@napi-rs/ffmpeg` or similar). Bundles ~5 MB per platform; dev-only install OK via `bun install`.
- (worst) ilink uses a proprietary format (e.g. SILK) that off-the-shelf TTS doesn't produce → either (a) find a JS SILK encoder, (b) ship audio as file attachment only (degraded UX).

Spike 4 outcome feeds back into this spec before implementation starts.

## Open questions (not blocking spec approval)

1. **Qwen "Cherry" voice vs. other options** — does DashScope's current API allow passing `voice` as a parameter, or is voice locked to the model id? Research during implementation.
2. **OpenAI voice options** — `alloy` / `echo` / `fable` / `onyx` / `nova` / `shimmer`. Which as default? Proposal: `nova` (female, warm, multi-language). Open during implementation.
3. **Rate limits** — Qwen DashScope has per-minute limits. Implementation should surface 429 as `{ok:false, reason:'rate_limited'}` so Claude can fall back to text without retry-looping.
4. **Cost budgeting** — do we want a per-day / per-turn cap (e.g. max 50 voice replies/day before refusing)? Not in scope for v1.1; can add later if abuse emerges.

## Acceptance criteria for v1.1 voice subset

- [ ] User sends a text message containing "念一下 X" → Claude produces audio in WeChat (verified on Windows E2E, matching Task 16 methodology).
- [ ] First-time user (no config) is walked through API-key registration via Claude conversation, saves, validates, and gets their voice reply — all within a single WeChat session.
- [ ] `reply_voice` with text > 500 chars gracefully falls back to `reply` with a short note.
- [ ] TTS provider 5xx does not hang the session; Claude falls back to text.
- [ ] `voice-config.json` is chmod 0600 on POSIX (documented no-op on Windows; fs.chmodSync still works but is weaker).
- [ ] Spike 4 outcome is documented in the release notes.
