# Spec · v1.2 Reliability Polish

**Status**: Draft · 2026-04-24
**Parent**: RFC 02 §3
**Expected effort**: 2-3 weeks

---

## Motivation

今晚的 session 暴露了 6 个字面上在跑但不健康的问题；多数被我们当场打了补丁，但补丁是止血不是根治。v1.2 把根治动作集中做掉，**不引入新功能**，目标是"重启 / 部署 / 断网 / session 过期，都不悄无声息"。

**范围**：`accounts/` 目录里 `8ca10d158998-im-bot` 和 `b1188b8b0251-im-bot` 两个 session 已过期 >48h，不会再收任何消息。删目录。

**动作**：
- `rm -rf ~/.claude/channels/wechat/accounts/{8ca10d158998,b1188b8b0251}-im-bot`
- daemon SIGUSR1 不会自动注销（reconcile 只 add，不 remove）— 需要重启 daemon
- 可选：把 remove 能力也加到 poll-loop handle（`stopAccount(id)`），reconcile 时发现目录消失就停 loop

**测试**：重启 daemon 后 `accounts=3`；reply 不再路由到死 bot。

**文件**：`src/daemon/poll-loop.ts` （补 `stopAccount`），`src/daemon/main.ts`（reconcile 处理删除）

## Task 2 · errcode 可见化 + session-expired 告警

**范围**：任何 ilink 返回 `errcode != 0` 都过 `assertIlinkOk`，但 `getupdates` 路径绕开了（`ilinkGetUpdates` 直接 JSON.parse 返 obj，不 assert）。

**动作**：
- `src/daemon/ilink-glue.ts` 的 `makeIlinkAdapter.ilink.getUpdates` 包装层加 errcode 检查
- errcode=-14 专门识别 → mark 该 account 为 `session_expired`
- 按 account 限流：每 bot 每 24h 最多推一次 admin 告警（避免刷屏）
- 告警文本走 reply 到 admins[0]：`⚠️ bot <id> session 过期，运行 wechat-cc setup 重扫`

**测试**：mock getupdates 返 errcode=-14，断言：
- [ ] `[ERROR]` log 出现，不是 silent
- [ ] admin 一次告警发出
- [ ] poll-loop 暂停该 account
- [ ] 同一 bot 重复 -14 不会刷告警

**文件**：`src/daemon/ilink-glue.ts`（adapter 层），`src/daemon/poll-loop.ts`（loop 停止逻辑）

## Task 3 · MCP 工具拆包（兑现 Spike 6）

**现状**：`src/features/tools.ts` = 397 行 / **22 个 tool**。即便用了 preset+append 绕开 ToolSearch 延迟加载，22 个工具进 system prompt 仍吃 context（粗估 ~3-4k tokens）+ Claude 决策树更宽。

**分组**（按职能 + tools.ts 现有常量顺序）：

| MCP server | 工具 | 何时装载 |
|---|---|---|
| `wechat-core` (6) | reply, edit_message, set_user_name, send_file, broadcast, _diagnostic | 常开 |
| `wechat-project` (4) | list_projects, switch_project, add_project, remove_project | 常开 |
| `wechat-docs` (2) | share_page, resurface_page | 常开 |
| `wechat-voice` (3) | reply_voice, save_voice_config, voice_config_status | **按需**（voice-config.json 存在才 attach） |
| `wechat-companion` (8) | enable/disable/status/snooze + persona_switch + trigger_{add,remove,pause} | **按需**（companion.enabled=true 才 attach） |

**实现**：`buildWechatMcpServer` 拆成 5 个 factory；bootstrap 按 feature flag 组装 `mcpServers: {core, project, docs, voice?, companion?}`。

**向后兼容**：当前 Claude 用 `mcp__wechat__*` 前缀引用工具；拆包后变成 `mcp__wechat-core__reply`。**会中断 Claude Code 现有 session 的工具熟悉度**，需要：
- feature flag `WECHAT_MCP_SPLIT=1` 默认 off 一周（灰度）
- 默认 on 之前跑一次 eval（Spike 6 原始目的）对比工具调用正确率

**测试**：
- [ ] 5 个 MCP server 都能独立起并响应 list_tools
- [ ] 关 voice-config.json → 只剩 4 个 server（voice 不 attach）
- [ ] 关 Companion → 只剩 3 个 server（companion 不 attach）
- [ ] 混合调用不同 server 的工具一条消息里正常（如 reply + switch_project）

**文件**：`src/features/tools.ts` 拆成 `src/features/{core,project,docs,voice,companion}-tools.ts`；`src/daemon/bootstrap.ts` 条件 attach。

## Task 4 · session 持久化 + resume

**现状**：`src/core/session-manager.ts` 用 in-memory `Map<alias, Internal>`。daemon 重启 → Map 空 → 每个 alias 首条消息走 SDK 新 session（~11s 冷启动 per Spike 1 data）。

**目标**：重启后仍能在 SDK `resume: <session_id>` 上继续，首 token <3s。

**设计**：
1. SessionManager 在 `onResult` callback 里记录 session_id（handle 已有 `onResult` hook）
2. 持久化到 `~/.claude/channels/wechat/sessions.json` — `{ [alias]: { session_id, last_used_at } }`（debounce 500ms 写入，跟其它 state 一样）
3. startup 时 SessionManager 读盘；`spawn(alias)` 时，如果有 recent session_id → 传 `resume: session_id` 给 `query()`
4. 如果 resume 失败（session 不在 SDK 里 / jsonl 损坏），fallback 到 `continue: true` 或全新 session

**Edge case**：
- 如果 user 通过 `claude --continue` 在同 cwd 自己开了会话 → SDK `continue` 会跟 daemon 争话头。用 resume 避开。
- Session 太老（>7 天）直接丢，全新起。

**测试**：
- [ ] 发一条 → result session_id 被 persist
- [ ] 重启 daemon → 同 chat 发第二条 → 首 token <3s（本地 Opus，不含网络）
- [ ] 手动删 sessions.json → fallback 到 fresh session，不崩

**文件**：`src/core/session-manager.ts` + 新增 `sessions.json` 管理（可复用 state-store pattern）

## Task 6 · Codex hook 配置修正（不是 wechat-cc 但今晚暴露）

**现状**：`/codex:setup` 某时配置了 `model: gpt-5.5`（不存在）；stop review gate 每次 session 结束都起一个 codex task → stream disconnected → 失败 → hook 反复报"task still running"。

**动作**：
- 手动跑 `/codex:setup` 把 model 改成 `gpt-5` 或 `gpt-5-codex`
- 或者直接把 `stopReviewGate=false` 永久关掉（`~/.claude/plugins/data/codex-openai-codex/state/<project>/state.json`），手动触发 review 而非自动
- 记 memory 提醒以后别开回去

**责任**：user action（我没法通过代码改 codex 插件内部状态）

## 验收标准（v1.2 整体）

v1.2 打 tag 前必须满足：

1. 重启 daemon 后首条消息 <3s 首 token（Task 4）
2. mock errcode=-14 场景触发 admin 告警（Task 2）
3. 5 个 MCP server 独立起（Task 3）+ 老 22-tool 兼容模式还能切回（feature flag）
4. `ilink-glue.ts` ≤200 行（Task 5），整体 LOC 不增加（拆分不膨胀）
5. 死 bot 清理工具跑一次，`accounts/` 只剩活 bot（Task 1）
6. Codex hook 不再报僵尸 task（Task 6）
7. 所有单元测试通过 + 新增的 resume / reconcile / errcode 测试齐全
