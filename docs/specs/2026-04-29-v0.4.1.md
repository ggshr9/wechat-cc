# v0.4.1 · SDK 接入与活动追踪

**Status**: design · 2026-04-29
**Tracks**: v0.4 留下的 3 项 deferred 工作

## 1 · 背景

v0.4 ship 的是基础设施 + UI；三件被显式 defer 的事都集中在「真 SDK 调用」这条线：

1. introspect 决策——目前 stub agent 永远返回 `{write: false, reasoning: 'v0.4.1 follow-up'}`。事件流通了，但没有真观察被写下
2. per-project 摘要——helpers (`needsRefresh` / `formatSummaryRequest`) 完整，但没人调；项目卡片永远显示 `(总结待 v0.4.1 上线)`
3. `ms_7day_streak` milestone——detector spec 已定义，但 `daysWithMessage` 永远是空数组，所以这条永不 fire

v0.4.1 把这三件交付，让 v0.4 的"承诺"和"行为"完全对齐。

## 2 · 目标

### 2.1 真 introspect SDK 集成

替换 `src/daemon/companion/introspect-runtime.ts::makeIntrospectAgent()` 的 stub 实现：

- **Spawn 隔离 SDK session**——不复用任何项目 session，避免 introspect 的"自审"风格污染用户的项目对话
- **Curated prompt**：注入 chat 当前 memory 内容 + 最近 N 条 events.jsonl 决策 + 最近 K 条 active observations + 最近 M 条入站消息（节流 prompt 长度上限 ~3000 token）
- **Structured output**：prompt 明确要求 Claude 返回一个 JSON `{write: bool, body?: string, tone?: 'concern'|'curious'|'proud'|'playful'|'quiet', reasoning: string}`。SDK 端 `JSON.parse` + 容错（如果不是合法 JSON，落到 `cron_eval_failed`）
- **Model**: `claude-haiku-4-5`——introspect 不需要顶级推理，成本/延迟优势明显
- **失败语义**：
  - SDK 超时/错误 → `events.append({kind: 'cron_eval_failed', trigger: 'introspect', reasoning: '<error message>'})`
  - JSON 解析失败 → 同上
  - 不写观察、不污染状态、不影响下一次 tick

### 2.2 真 per-project LLM 摘要

- **触发时机**：lazy。当 CLI `sessions list-projects` 被调用时（dashboard 打开会话 pane 触发），检查每个 project 的 `needsRefresh()`；为 stale 的项目**异步 fire-and-forget** 一次摘要 eval。当前请求立即返回旧值（或 null），下次刷新会看到新摘要
- **隔离 SDK session**——同 introspect 同样的隔离原则
- **Curated prompt**：`formatSummaryRequest(turns)`——读项目 jsonl 末尾 20 条 turn，拼成对话片段，问 Claude "用一句话（中文，≤30 字）总结最后做了什么"
- **Output**：纯文本（一行）。Trim、去掉前后引号/空格、上限 50 字（截断保护）
- **Model**: `claude-haiku-4-5`
- **写回**：新增 `SessionStore.setSummary(alias, summary)` API，写 `summary` + `summary_updated_at` 到 `sessions.json`
- **失败语义**：
  - SDK 失败 / 输出空 → 不更新 sessions.json（保留旧值或 null）
  - 不重试本次；下次 list 命中 stale 时再触发

### 2.3 `ms_7day_streak` 真落地

- **新文件**：`memory/<chat_id>/activity.jsonl`，schema:
  ```jsonl
  {"date":"2026-04-29","first_msg_ts":"...ISO...","msg_count":12}
  ```
  Per-day 一条记录。`date` 是该 chat 的本地日历日（YYYY-MM-DD）；`first_msg_ts` 是当天第一条入站消息的 ISO 时间；`msg_count` 是当天累计入站次数
- **写入时机**：daemon `onInbound` 收到消息时，检查今天的 record 是否已存在（按 date），不存在就追加一条；存在就 `msg_count += 1`（rewrite-with-rename，跟 observations 一样的 atomic 模式）
- **TTL 30 天**：read path 过滤；rewrite 时也可以 prune，但留给 v0.5 做（先简单实现）
- **detector 接入**：`buildDetectorContext` 读 activity.jsonl，filter 最近 7 天，传给 detector 的 `daysWithMessage`
- **新增 events kind**：n/a——streak milestone 触发后走现有的 `kind: 'milestone'` 路径，跟 100/1000-msg 走同一条线

### 2.4 新增 events kind: `cron_eval_failed`

为 introspect 和 summarizer 共用：

```ts
export type EventKind =
  | 'cron_eval_pushed'
  | 'cron_eval_skipped'
  | 'cron_eval_failed'      // NEW
  | 'observation_written'
  | 'milestone'
```

decisions log UI（决策日志面板）也展示这条——glyph 用 `⚠`，summary "Claude 想 introspect 但 SDK 出错了"。

## 3 · 架构决定（已 ratify）

| | 决定 | 理由 |
|---|---|---|
| A | introspect/summarizer 用**隔离 SDK session** | 避免 system prompt 风格污染项目对话；隔离失败也不影响主链路 |
| B | output format = **prompt-engineer + JSON.parse** | tool_call schema 维护成本高；纯 prompt + 容错足够稳 |
| C | model = **claude-haiku-4-5** for both | 成本/延迟优；introspect/摘要任务复杂度不需要 Sonnet 起步 |
| D | 失败 → `cron_eval_failed` 事件 | 给 dashboard "决策日志"留可见痕迹，便于诊断 |
| E | activity.jsonl 一日一条 | schema 简单、文件大小可控；7-day-streak 检查只读最近 7 行 |

## 4 · 不在 scope 内（v0.5+）

- 多 chat 切换的 dashboard chat-picker
- activity.jsonl 自动 prune（v0.5 加）
- introspect 频率自适应（如"用户活跃时降频"）
- 摘要的 SQLite FTS / 跨项目索引
- introspect 写入失败的重试退避（目前是单次 attempt，跳过即过）

## 5 · 风险与回滚

| 风险 | 概率 | 兜底 |
|---|---|---|
| Claude 写"创飞"的观察（错误推断、冒犯）| 中 | 用户可一键 archive；prompt 里加 "也许你看错了" 风格 guidance；observations 写入前可加关键词自检（v0.5） |
| introspect cron 资源消耗（每天 1 次 SDK eval × Haiku 成本）| 低 | jitter + 跳过空 chat（30 天无消息）+ 失败不影响主链路 |
| 摘要 lazy fire-and-forget 同时多个 stale project | 低-中 | 进程内 in-memory queue 限制并发为 1；sessions.json 写入 debounced |
| activity.jsonl 在大量入站时的 race | 低 | atomic rewrite + 单进程假设（同 observations 的并发 posture） |
| Haiku 偶尔输出非 JSON（hallucinate prefix/suffix）| 中 | 解析时容错：strip 三个反引号块、找 first `{`、最后一个 `}`、只解析中间 |

## 6 · 验收

- 用户首次使用 1 周后，记忆 pane 顶部能看到至少 1-2 条 Claude **真**写下的观察（非 demo seed）
- 会话 pane 项目卡片显示真 LLM 一行摘要（"修了 ilink-glue" 这种），不再是 `(总结待 v0.4.1)`
- 连续 7 天每天有消息后，里程碑卡片出现 "我们已经连续 7 天每天都聊"
- 决策日志在 SDK 偶尔失败时能看到 ⚠ 行，原因可读
